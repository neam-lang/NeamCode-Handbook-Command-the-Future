Chapter 3: Talking to NeamCode — The Art of Prompting #

"The quality of an answer is determined by the quality of the question."

📖 20 min read

What you'll learn:

  • How to write prompts that get great results
  • The difference between vague and specific requests
  • Multi-step prompting strategies
  • When to give context vs. when to let NeamCode explore
  • Common prompting mistakes and how to avoid them

3.1 NeamCode Isn't a Search Engine #

The biggest mistake new users make is treating NeamCode like Google. They type short, vague queries and expect magic:

text
❌ Bad:  "fix bug"
❌ Bad:  "make it work"
❌ Bad:  "help"

NeamCode is an agent. It takes actions. The better you describe what you want, the better it performs. Think of it like talking to a skilled colleague who just joined your team — they're brilliant, but they don't know your project yet.

text
✅ Good: "The /api/users endpoint returns 500 when the email field
         is missing from the request body. Find and fix the issue."

✅ Good: "Create a React component called UserCard that shows the
         user's name, avatar, and role. Use Tailwind CSS for styling."

✅ Good: "I just cloned this repo. Give me a high-level overview of
         the project structure, the main entry points, and how data
         flows through the system."

3.2 The Prompting Spectrum #

Think of prompts on a spectrum from open to specific:

text
Open ◄──────────────────────────────────► Specific

"Explore this      "What does the        "In auth.ts line 42,
 codebase"          login function        change the bcrypt
                    do?"                  salt rounds from
                                          10 to 12"

Both ends of the spectrum are valid. Use them for different situations:

Situation Prompt Style Example
You don't know the codebase Open "Help me understand this project"
You know what's wrong but not where Medium "Find where user sessions expire"
You know exactly what to change Specific "In config.ts, change the timeout from 30 to 60 seconds"
You want ideas Open "How could we improve the error handling here?"
You want implementation Specific "Add try-catch blocks to all the async functions in api/"
💡 Tip

💡 Tip — When in doubt, start open and narrow down. NeamCode can explore, and you can guide it based on what it finds.

3.3 Six Prompt Patterns That Work #

Pattern 1: Describe the Problem, Not the Solution #

Instead of telling NeamCode how to fix something, describe what's wrong:

text
❌ "Add a null check on line 47 of user-service.ts"

✅ "The app crashes when a user has no profile picture.
    Find where the crash happens and fix it."

The second version lets NeamCode investigate. It might find that the issue is actually on line 52, or in a different file entirely.

Pattern 2: Give the "Why" Along with the "What" #

Context makes NeamCode smarter:

text
❌ "Convert this function to TypeScript"

✅ "Convert this function to TypeScript. We're migrating from
    JavaScript and want strict types. Follow the patterns used
    in the rest of the src/services/ directory."

Pattern 3: Use Examples for Style #

If you want output in a particular style, show an example:

text
You: Write unit tests for the calculateTotal function.
     Follow this pattern from our existing tests:

     test('returns 0 for empty cart', () => {
       expect(calculateTotal([])).toBe(0);
     });

Pattern 4: Step-by-Step for Complex Tasks #

For large tasks, break them into steps — or ask NeamCode to plan first:

text
You: I want to add Google OAuth to this Express app.
     Before writing any code, outline the steps you'd take.
     Then implement them one at a time.

Pattern 5: Constrain the Scope #

Prevent NeamCode from going too far:

text
You: Refactor the database queries in user-repo.ts to use
     parameterized queries. Only change that file — don't
     touch anything else.

Pattern 6: Ask for Verification #

Tell NeamCode to check its own work:

text
You: Fix the failing test in auth.test.ts. After fixing it,
     run the test suite to make sure everything passes.

3.4 Multi-Turn Conversations #

NeamCode remembers everything in your session. Use this to build up complex work:

text
Turn 1: "Show me the project structure"
         → NeamCode explores and describes the codebase

Turn 2: "Which files handle authentication?"
         → NeamCode searches and identifies the auth files

Turn 3: "The JWT token expiry is too short. Find where it's set."
         → NeamCode reads the auth files and finds the config

Turn 4: "Change it from 1 hour to 24 hours"
         → NeamCode edits the exact line

Turn 5: "Are there any tests for this? Run them."
         → NeamCode finds and runs the relevant tests

Each turn builds on the last. You don't need to re-explain the context.

⚠️ Warning — Very long conversations (100+ turns) can slow down NeamCode as the context window fills up. If things get sluggish, use /compact to summarize the conversation, or start a new session and use /memory to carry forward what matters. See Chapter 10 for details.

3.5 When to Give Context vs. When to Let It Explore #

Let NeamCode Explore When: #

  • You're new to the codebase
  • You're not sure where a bug is
  • You want a fresh perspective
  • The task involves multiple files
text
You: I'm getting "Cannot read property 'id' of undefined"
     somewhere in the checkout flow. Find it.

NeamCode will search for the error, trace the data flow, and find the root cause — often discovering things you wouldn't have found manually.

Give Context When: #

  • You know exactly which file to change
  • You have specific requirements (framework, style, patterns)
  • You want to prevent unnecessary exploration
  • You're doing something unusual
text
You: In src/middleware/auth.ts, the verifyToken function uses
     the jsonwebtoken library. Replace it with jose because
     we're moving to edge runtime which doesn't support
     jsonwebtoken's native bindings.

3.6 Prompting for Different Tasks #

Understanding Code #

text
You: Explain the data flow from when a user clicks "Submit Order"
     to when the order appears in the database. Trace through
     every function call.

Writing New Code #

text
You: Create a REST API endpoint POST /api/notes that accepts
     { title: string, content: string } and saves it to the
     notes table. Include input validation and error handling.
     Follow the patterns in the existing endpoints.

Debugging #

text
You: The test in payment.test.ts on line 45 is failing with
     "expected 99.99 but received 100". Help me understand
     why and fix it.

Refactoring #

text
You: The UserService class in user-service.ts has grown to
     400 lines. Split it into smaller, focused modules while
     keeping the public API the same. Don't break any tests.

Learning #

text
You: I'm not familiar with React hooks. Explain how useEffect
     works and show me examples from this project's codebase.

3.7 Common Mistakes #

Mistake 1: Being Too Vague #

text
❌ "Make the app better"
✅ "Improve the error messages in the validation layer so users
    know exactly which field is invalid and why"

Mistake 2: Giving Too Many Instructions at Once #

text
❌ "Add authentication, set up a database, write tests, create
    a CI pipeline, deploy to AWS, and add monitoring"

✅ "Let's add authentication. Start by setting up JWT-based
    auth with a login endpoint."
(Then continue with more requests in subsequent turns)

Mistake 3: Not Reading What NeamCode Says #

NeamCode often explains what it found, asks clarifying questions, or warns about potential issues. Read its responses — they contain valuable information.

Mistake 4: Fighting the Permission System #

If NeamCode asks permission to do something and you keep denying it, it can't help you. Either approve the action or explain what you want differently:

text
NeamCode: I'd like to run: npm install express
Allow? (y/n)

❌ n (then wondering why it doesn't work)
✅ y (let it install the dependency)
✅ "Don't install anything. Just show me the code and I'll
    install manually." (give clear alternative instructions)

3.8 Pro Tips #

Tip 1: Use Natural Language First #

Don't try to speak in commands or code. NeamCode understands natural language best:

text
❌ "grep -r 'TODO' src/ | wc -l"
✅ "How many TODO comments are in the src directory?"

Tip 2: Reference Files by Name #

If you know the file, mention it. NeamCode will read it immediately:

text
"Look at package.json and tell me which dependencies
 are outdated."

Tip 3: Say "Don't" When Needed #

If you want to prevent certain actions:

text
"Refactor the API routes but don't change any test files."
"Fix the CSS but don't use inline styles."
"Add error handling but don't catch and silently ignore errors."

Tip 4: Ask NeamCode to Think First #

For complex tasks, ask it to plan before acting:

text
"Before making any changes, analyze the current architecture
 and propose your approach. I'll confirm before you proceed."

Exercises #

🎯 Exercise 3.1 — Open NeamCode in any project directory. Try these three prompts and compare the results:

  1. "Tell me about this project"
  2. "What does this project do, what technologies does it use, and what's the directory structure?"
  3. "Read the README and package.json, then give me a 5-bullet summary of this project's purpose, tech stack, and entry points."

🎯 Exercise 3.2 — Think of a bug you recently fixed. Write a prompt describing the symptoms (not the solution) and see if NeamCode can find and fix it.

🎯 Exercise 3.3 — Ask NeamCode to create a small project (e.g., a Python CLI tool, a Node.js web server, or a bash utility script). Start with an open prompt, then refine with follow-up turns.

🎯 Exercise 3.4 — Practice the "plan first" pattern: ask NeamCode to outline its approach for adding a feature, then approve or modify the plan before it starts coding.
