Chapter 9: Modes — Different Hats for Different Tasks #

"It is not the strongest of the species that survives, nor the most intelligent, but the one most adaptable to change."
— Charles Darwin

📖 Estimated read time: 22 minutes

What You'll Learn #

  • What modes are and why they matter
  • How each of the 6 modes changes NeamCode's behavior
  • Real terminal examples of every mode in action
  • When to use which mode for maximum effectiveness
  • How to combine modes within a single work session
  • Power-user tips for mode-based workflows

9.1 What Are Modes? #

Modes are like different hats you put on NeamCode. When you switch modes, you change how NeamCode thinks, what it prioritizes, and what tools it reaches for first.

Think of it this way:

text
┌─────────────────────────────────────────────────────┐
│              Same NeamCode, Different Focus          │
│                                                     │
│   You ──→ /mode explore ──→ "Let me read and map    │
│                              this codebase for you"  │
│                                                     │
│   You ──→ /mode debug   ──→ "Show me the error,     │
│                              let's trace it"         │
│                                                     │
│   You ──→ /mode deploy  ──→ "Let's get this live     │
│                              and running"            │
│                                                     │
└─────────────────────────────────────────────────────┘

When you switch modes, NeamCode adjusts:

  • System prompt — Its underlying instructions shift to match the task
  • Tool preferences — It reaches for different tools first
  • Response style — More explanatory in Explore, more action-oriented in Deploy
  • Suggestions — It proactively offers mode-relevant next steps

Switching is instant, free, and you can do it as many times as you want in a session:

text
neamcode> /mode debug

  ✓ Switched to Debug mode
  Debug mode active — I'll focus on finding and fixing bugs.
  Paste an error message or describe the issue.

9.2 Explore Mode #

Purpose: Understand codebases you didn't write (or wrote six months ago and forgot).

Explore mode is the default when you start NeamCode. It's optimized for reading, mapping, and explaining code.

What Changes in Explore Mode #

  • NeamCode reads more files before answering
  • Responses include architecture context and dependency explanations
  • It draws connections between components
  • It avoids modifying code unless explicitly asked

Explore Mode in Action #

text
neamcode> /mode explore

  ✓ Switched to Explore mode
  I'll focus on understanding and explaining code.

neamcode> How is authentication handled in this project?

  I've examined the codebase and here's how authentication works:

  Authentication Flow
  ═══════════════════

  ┌──────────┐    ┌──────────────┐    ┌──────────────┐
  │  Client   │───→│  Auth        │───→│  Route       │
  │  Request  │    │  Middleware   │    │  Handler     │
  └──────────┘    └──────┬───────┘    └──────────────┘
                         │
                  ┌──────▼───────┐
                  │  Clerk SDK    │
                  │  (clerk.com)  │
                  └──────────────┘

  1. **Entry point:** `src/middleware/auth.ts` (line 12)
     - Extracts the Bearer token from the Authorization header
     - Calls Clerk's `verifyToken()` to validate

  2. **Session handling:** `src/middleware/session.ts`
     - Attaches the decoded user object to `req.user`
     - Sets session cookie with 24-hour expiry

  3. **Route protection:** Routes use `requireAuth` middleware
     - Public routes: `/api/health`, `/api/auth/login`
     - Protected routes: Everything under `/api/users/*`

  4. **Role-based access:** `src/middleware/rbac.ts`
     - Three roles: admin, editor, viewer
     - Checked via `requireRole('admin')` middleware

  Key files:
  • src/middleware/auth.ts
  • src/middleware/session.ts
  • src/middleware/rbac.ts
  • src/config/clerk.ts

When to Use Explore Mode #

  • Onboarding to a new codebase
  • Understanding how a feature works end-to-end
  • Finding where something is defined or used
  • Getting architecture overviews
  • Reviewing someone else's code for comprehension
💡 Tip

💡 Tip: Explore mode is great for the question "Where would I even start?" Ask NeamCode to map the entry points and data flow for any feature.

9.3 Debug Mode #

Purpose: Find bugs, trace errors, and generate fixes.

Debug mode turns NeamCode into a diagnostic partner. It asks more targeted questions, traces execution paths, and proposes fixes with explanations.

What Changes in Debug Mode #

  • NeamCode prioritizes error analysis and root-cause investigation
  • It looks at stack traces, logs, and error messages first
  • Responses include specific fix suggestions with file/line references
  • It proactively checks for common related issues

Debug Mode in Action #

text
neamcode> /mode debug

  ✓ Switched to Debug mode
  Paste an error message or describe the issue.

neamcode> I'm getting "TypeError: Cannot read properties of undefined
  (reading 'id')" in the user profile page

  Let me trace this error.

  Investigating...
  ════════════════

  I found the issue in `src/pages/Profile.tsx` at line 34:

// Line 34 — the bug const userId = session.user.id;

text

  The problem: `session.user` can be `undefined` when the page
  loads before the auth session is fully initialized.

  Root Cause
  ──────────
  The `useSession()` hook from Clerk returns `{ session: null }`
  during the initial render. Your component doesn't guard against
  this.

  Fix
  ───

// Option 1: Early return with loading state const { session, isLoaded } = useSession();

if (!isLoaded || !session?.user) { return <LoadingSpinner />; }

const userId = session.user.id; // Now safe

text

  Related Issues I Noticed
  ────────────────────────
  ⚠ The same pattern exists in `src/pages/Settings.tsx` (line 22)
    and `src/pages/Dashboard.tsx` (line 18). You may want to fix
    those too.

  Want me to apply this fix to all three files?

Paste-and-Fix Workflow #

Debug mode excels at the paste-an-error workflow:

text
neamcode> npm run build

  ERROR in src/api/routes.ts:47:5
  TS2345: Argument of type 'string' is not assignable to
  parameter of type 'number'.

  ─────────────────────────────────

  The issue is on line 47 of `src/api/routes.ts`:

const result = fetchUser(req.params.id); // ^^^^^^^^^^^^^ // req.params.id is a string, but fetchUser expects a number

text

  Fix:

const result = fetchUser(parseInt(req.params.id, 10));

text

  Or better, with validation:

const id = parseInt(req.params.id, 10); if (isNaN(id)) { return res.status(400).json({ error: 'Invalid user ID' }); } const result = fetchUser(id);

text

  Want me to apply the safer version?
⚠️ Warning

⚠️ Warning: Debug mode is more action-oriented than Explore. It may suggest code changes proactively. Always review before accepting.

9.4 Test Mode #

Purpose: Write tests, run test suites, and improve coverage.

Test mode is optimized for generating test cases, running test commands, and analyzing coverage reports.

What Changes in Test Mode #

  • NeamCode generates test code that matches your project's testing framework
  • It focuses on edge cases and boundary conditions
  • It can run tests and interpret results
  • It suggests what else to test

Test Mode in Action #

text
neamcode> /mode test

  ✓ Switched to Test mode
  I'll help you write and run tests. What would you like to test?

  I've analyzed `fetchUser` and here are comprehensive tests:

9.5 Analyze Mode #

Purpose: Work with data — query databases, transform datasets, create visualizations.

Analyze mode is built for data work. It's comfortable with SQL, data pipelines, CSV files, and statistical analysis.

What Changes in Analyze Mode #

  • NeamCode prioritizes data exploration and querying
  • It generates SQL, pandas code, or data transformation scripts
  • It focuses on patterns, anomalies, and insights
  • It can work with MCP-connected databases directly

9.6 Deploy Mode #

Purpose: Build, ship, and monitor your applications.

Deploy mode focuses on infrastructure, CI/CD, containerization, and cloud services. It's action-oriented and safety-conscious.

What Changes in Deploy Mode #

  • NeamCode focuses on infrastructure files (Dockerfiles, configs, CI/CD)
  • It double-checks destructive operations
  • It generates deployment scripts with rollback plans
  • It proactively flags security and cost concerns

Deploy Mode in Action #

text
neamcode> /mode deploy

  ✓ Switched to Deploy mode
  I'll help with infrastructure and deployment tasks.
  What are we shipping?

neamcode> Help me containerize this Node.js app

  I'll create a production-ready Dockerfile.
  ════════════════════════════════════════════

  Analyzing your project...
  • Runtime: Node.js 20 (from .nvmrc)
  • Package manager: pnpm (from pnpm-lock.yaml)
  • Build step: `pnpm build` outputs to `dist/`
  • Port: 3000 (from src/server.ts)

  Creating Dockerfile:

# Build stage FROM node:20-alpine AS builder WORKDIR /app RUN corepack enable && corepack prepare pnpm@latest --activate COPY package.json pnpm-lock.yaml ./ RUN pnpm install --frozen-lockfile COPY . . RUN pnpm build

# Production stage FROM node:20-alpine AS runner WORKDIR /app RUN addgroup -g 1001 -S appgroup && \ adduser -S appuser -u 1001 COPY --from=builder /app/dist ./dist COPY --from=builder /app/node_modules ./node_modules COPY --from=builder /app/package.json ./ USER appuser EXPOSE 3000 HEALTHCHECK --interval=30s --timeout=3s \ CMD wget --spider http://localhost:3000/health || exit 1 CMD ["node", "dist/server.js"]

text

  Also creating .dockerignore:

node_modules .git .env *.md tests/

text

  ⚠ Security notes:
  • Runs as non-root user (appuser)
  • Uses multi-stage build to minimize image size
  • No secrets baked into the image

  Want me to also create a docker-compose.yml for local development?
⚠️ Warning

⚠️ Warning: Deploy mode will ask for confirmation before running commands that affect live infrastructure. Always double-check the target environment.

9.7 Ideate Mode #

Purpose: Brainstorm, explore ideas, and design solutions before writing code.

Ideate mode is the creative mode. It's less concerned with exact implementations and more focused on generating options, weighing trade-offs, and exploring possibilities.

What Changes in Ideate Mode #

  • NeamCode generates multiple alternatives instead of one answer
  • It weighs pros and cons explicitly
  • It asks clarifying questions about goals and constraints
  • It's more conversational and exploratory

Ideate Mode in Action #

text
neamcode> /mode ideate

  ✓ Switched to Ideate mode
  Let's brainstorm. What problem are we solving?

neamcode> We need to add real-time notifications to our app.
  Currently we have a REST API with PostgreSQL.

  Great question to explore! Let me lay out the main approaches:

  Option 1: WebSockets (Socket.IO)
  ═════════════════════════════════
  ✓ True real-time, bidirectional
  ✓ Mature ecosystem (Socket.IO)
  ✗ Requires persistent connections (more server resources)
  ✗ Need to handle reconnection logic
  Best for: Chat-like features, live dashboards

  Option 2: Server-Sent Events (SSE)
  ═══════════════════════════════════
  ✓ Simpler than WebSockets
  ✓ Works over standard HTTP
  ✓ Auto-reconnection built into browsers
  ✗ One-directional (server → client only)
  ✗ Limited browser connection pool
  Best for: Notification feeds, status updates

  Option 3: Polling (Short or Long)
  ══════════════════════════════════
  ✓ Simplest to implement
  ✓ Works everywhere, no special infrastructure
  ✗ Not truly real-time (delay between polls)
  ✗ Wasteful — constant requests even when nothing changes
  Best for: Low-frequency updates, MVP/prototype

  Option 4: PostgreSQL LISTEN/NOTIFY + SSE
  ═════════════════════════════════════════
  ✓ Uses your existing database
  ✓ No additional infrastructure needed
  ✓ Triggers on actual data changes
  ✗ Doesn't scale past ~10K concurrent listeners
  Best for: Your current stack — minimal new dependencies

  My Recommendation
  ─────────────────
  Given your current REST + PostgreSQL stack, I'd suggest
  starting with **Option 4** (PG LISTEN/NOTIFY + SSE):

  ┌──────────┐   NOTIFY   ┌──────────┐   SSE    ┌──────────┐
  │ Database  │──────────→ │  API     │────────→ │  Client  │
  │ (trigger) │            │  Server  │          │  Browser  │
  └──────────┘            └──────────┘          └──────────┘

  It requires zero new infrastructure and gives you real-time
  updates for your current scale. If you outgrow it, migrating
  to WebSockets later is straightforward.

  Want me to explore any of these options in more detail?
  Or should we think about the notification data model?
💡 Tip

💡 Tip: Ideate mode is perfect for the start of a project or feature. Use it to think through options before switching to Explore or Debug mode for implementation.

9.9 Plan Mode (v0.3.5.1) #

v0.3.5.1 introduces Plan Mode — a dedicated mode for designing implementation strategies before writing code. Unlike other modes that let NeamCode take action immediately, Plan Mode is read-only: it can read files and search code but cannot modify anything.

Entering Plan Mode #

text
neamcode> /plan Add rate limiting to all API endpoints

  📋 Entering Plan Mode...

  Analyzing codebase...
  [file_read] src/api/routes/index.ts
  [grep] "router\." in src/api/
  [find_symbol] rateLimit

  Plan: API Rate Limiting
  ═══════════════════════

  Step 1: Install express-rate-limit package
          Estimated: ~200 tokens

  Step 2: Create rate limit config (src/config/rate-limit.ts)
          - Default: 100 requests / 15 min per IP
          - Auth endpoints: 10 requests / 15 min
          - API endpoints: 60 requests / 1 min
          Estimated: ~800 tokens

  Step 3: Create rate limit middleware (src/middleware/rate-limit.ts)
          - Apply default limiter globally
          - Apply auth limiter to /auth/* routes
          - Custom key generator for authenticated users
          Estimated: ~1,200 tokens

  Step 4: Add Redis store for distributed rate limiting
          - Connect to existing Redis instance from config
          Estimated: ~600 tokens

  Step 5: Write tests (tests/rate-limit.test.ts)
          - Test default limits
          - Test auth endpoint limits
          - Test authenticated user key generation
          Estimated: ~1,500 tokens

  ─────────────────────────────────────────
  Total: 5 steps  |  ~4,300 tokens  |  ~$0.06
  Dependencies: express-rate-limit, rate-limit-redis

  [a]pprove  [e]dit  [c]ancel

Why Plan Mode Matters #

  • No accidental changes — Read-only mode prevents premature implementation
  • Cost estimation — See token and dollar costs before committing
  • Task decomposition — Complex tasks broken into clear, ordered steps with dependencies
  • Review before execution — You approve, edit, or cancel the plan before any code is written

Reasoning Strategies (v0.3.5.1) #

Plan Mode uses one of four reasoning strategies, selected automatically based on task complexity:

Strategy Best For How It Works
Chain of Thought Simple, linear tasks Step-by-step reasoning (1 LLM call)
Plan and Execute Multi-step implementations Decompose → plan → execute (N+2 calls)
Tree of Thought Architecture decisions Explore multiple paths (2-3 calls)
Self Consistency Critical/ambiguous tasks Multiple reasoning paths, majority vote (N+1 calls)

You can override the automatic selection:

text
neamcode> /think tot
  Reasoning strategy set to: Tree of Thought

neamcode> /plan Should we use GraphQL or REST for the new API?

  Branch A: GraphQL
    Pros: Flexible queries, single endpoint, type safety
    Cons: Complexity, caching difficulty, N+1 queries

  Branch B: REST
    Pros: Simple, cacheable, well-understood
    Cons: Over/under-fetching, multiple endpoints

  Branch C: Hybrid (REST + GraphQL for complex queries)
    Pros: Best of both, incremental adoption
    Cons: Two systems to maintain

  Recommendation: Branch B (REST) based on your codebase patterns
  and team experience. GraphQL adds complexity without sufficient
  benefit for your 8-endpoint API.

9.10 Pro Tips #

Mode Persistence #

Modes persist within a session. If you /resume a session, you return to whatever mode you were last using:

text
neamcode> /resume payment-fix

  ✓ Resumed session "payment-fix" (23 messages)
  Mode: Debug (restored from session)

The Default Mode #

When you start a new session without specifying a mode, NeamCode defaults to Explore mode. This is intentional — understanding comes before action.

Mode + Memory Interaction #

NeamCode's memory system (MEMORY.md) works across all modes. If you tell NeamCode to remember something in Debug mode, that knowledge is available in every other mode too:

text
neamcode> /mode debug
neamcode> Remember that the double-charge bug was caused by
  a missing idempotency key in the Stripe webhook handler

  ✓ Saved to MEMORY.md

neamcode> /mode explore
neamcode> What do you know about our payment issues?

  From memory: The double-charge bug was caused by a missing
  idempotency key in the Stripe webhook handler.

Don't Overthink Mode Selection #

Modes are helpers, not constraints. If you're in Explore mode and ask NeamCode to fix a bug, it will still help. Modes just optimize the approach — they don't limit capability.

If you find yourself constantly switching, you might be better off just staying in the mode closest to your primary task and letting NeamCode adapt.

Modes in Neam Programs #

If you're writing Neam agents, you can specify modes in your agent definitions for automated workflows:

neam
agent CodeReviewer {
  provider: "openai",
  model: "gpt-4o",
  mode: "explore",
  system: "Review pull requests for quality and correctness"
}
📝 Note

📝 Note: Modes affect the AI's behavior, not your permissions. You can always override a mode's tendencies with explicit instructions.
