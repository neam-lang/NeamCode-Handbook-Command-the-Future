24.1 Themes #

Setting a Persistent Theme #

Your theme choice is saved automatically and persists across sessions. You can also set it in your configuration file:

text
> /config set theme dracula

Configuration updated:
  theme: dracula (saved to ~/.neamcode/config.json)
💡 Tip

💡 Tip: If you work in different lighting conditions, set up a quick alias. For example, use solarized-light during the day and dracula at night. Some users create a shell alias: alias ncday='neamcode --theme solarized-light'.

24.2 Vim Mode #

For developers who think in Vim keybindings, NeamCode supports Vim mode for the input editor.

Enabling Vim Mode #

text
> /vim on

Vim mode enabled.
  - Normal mode: navigate and edit
  - Insert mode: type text
  - Visual mode: select text
  - Press ESC to return to normal mode
  - Press i to enter insert mode
  - Use :w to submit your message (same as Enter in normal mode)

Status bar now shows: [NORMAL] or [INSERT]

Supported Vim Commands #

text
Navigation:
  h, j, k, l    Move cursor left, down, up, right
  w, b           Move by word forward, backward
  0, $           Move to start, end of line
  gg, G          Move to start, end of input

Editing:
  i, a           Insert before/after cursor
  I, A           Insert at start/end of line
  o, O           New line below/above
  x              Delete character
  dd             Delete line
  dw             Delete word
  cc             Change line
  cw             Change word
  yy             Yank (copy) line
  p              Paste
  u              Undo
  Ctrl+r         Redo

Commands:
  :w             Submit message (like Enter)
  :q             Quit NeamCode
  :wq            Submit and then quit
  /pattern       Search in input

Disabling Vim Mode #

text
> /vim off

Vim mode disabled. Standard editing restored.
📝 Note

📝 Note: Vim mode only affects the input editor (where you type your messages). It does not change how you navigate NeamCode's output or menus. Slash commands still work normally in both modes.

24.3 Custom Keybindings #

You can remap keyboard shortcuts to match your workflow.

Viewing Current Keybindings #

text
> /keybindings list

Current keybindings:
  Ctrl+C         Cancel current operation
  Ctrl+D         Exit NeamCode
  Ctrl+L         Clear screen
  Ctrl+R         Search command history
  Ctrl+Z         Undo last edit
  Tab            Accept autocomplete suggestion
  Shift+Tab      Previous autocomplete suggestion
  Up/Down        Navigate command history
  Ctrl+Up        Scroll output up
  Ctrl+Down      Scroll output down
  Ctrl+S         Save current session
  Ctrl+K         Toggle compact mode

Customizing Keybindings #

text
> /keybindings set Ctrl+P "/review"

Keybinding set:
  Ctrl+P → runs /review command

> /keybindings set Ctrl+B "/mode debug"

Keybinding set:
  Ctrl+B → switches to debug mode

Removing a Keybinding #

text
> /keybindings remove Ctrl+P

Keybinding removed: Ctrl+P

> /keybindings reset

All keybindings reset to defaults.

Custom keybindings are saved to ~/.neamcode/keybindings.json and persist across sessions.

Celebrations #

When NeamCode completes a significant task (like finishing a code review or all tests passing), it can show a small celebration.

text

Celebrations enabled.

24.6 NEAMCODE.md Customization Patterns #

The NEAMCODE.md file in your project root is the most powerful customization tool. It gives NeamCode persistent context about your project, preferences, and conventions.

Basic Template #

Create a file called NEAMCODE.md in your project root:

markdown
# Project: My Application

## Stack
- Language: TypeScript 5.4
- Framework: Next.js 14 (App Router)
- Database: PostgreSQL 16
- ORM: Drizzle
- Testing: Vitest

## Code Conventions
- Use functional components, no classes
- Prefer named exports over default exports
- All functions must have JSDoc comments
- Error handling: use Result<T, E> pattern
- File naming: kebab-case (e.g., user-service.ts)

## Architecture
- src/app/ — Next.js app router pages
- src/components/ — React components
- src/lib/ — Shared utilities
- src/server/ — Server-side code
- src/db/ — Database schema and migrations

## Testing Rules
- Every new feature needs tests
- Minimum 80% coverage on changed files
- Use test factories, not raw fixtures
- Mock external APIs, never hit real endpoints in tests

Advanced Patterns #

Per-directory context:

You can create NEAMCODE.md files in subdirectories for specific context:

text
project/
├── NEAMCODE.md              # Project-wide context
├── src/
│   └── auth/
│       └── NEAMCODE.md      # Auth-specific context
└── tests/
    └── NEAMCODE.md          # Testing-specific context
markdown
# src/auth/NEAMCODE.md

## Auth Module Context
- We use Auth0 for authentication
- JWTs are validated in middleware, not in handlers
- User roles: admin, editor, viewer
- Session tokens expire in 24 hours
- NEVER store passwords — Auth0 handles this
- All auth endpoints require HTTPS

Personal preferences file:

Create ~/.neamcode/NEAMCODE.md for global preferences that apply to all projects:

markdown
# Personal NeamCode Preferences

## Communication Style
- Be concise — I prefer short explanations
- Show code first, explain after
- Use bullet points, not paragraphs
- Don't explain basic concepts I already know
- When I ask "why", give the technical reason, not a tutorial

## Defaults
- Always use TypeScript, never JavaScript
- Prefer functional patterns over OOP
- Use single quotes, not double quotes
- 2-space indentation
- Trailing commas in multi-line expressions

NEAMCODE.md Precedence #

When multiple context files exist, NeamCode merges them with this priority:

text
  Highest priority
    ↓
  1. Prompt-level instructions (what you type)
  2. Directory-level NEAMCODE.md (closest to current file)
  3. Project-root NEAMCODE.md
  4. Global ~/.neamcode/NEAMCODE.md
    ↓
  Lowest priority
📝 Note

📝 Note: Keep NEAMCODE.md files focused and concise. A 2,000-line context file wastes tokens and slows responses. Aim for under 100 lines per file.

24.7 Permission Tuning #

NeamCode has four permission modes that control how much autonomy it has.

Permission Modes Overview #

text
  ┌──────────────────────────────────────────────────────────┐
  │                 Permission Modes                          │
  ├───────────┬───────────────────────────────────────────────┤
  │   Mode    │ Behavior                                     │
  ├───────────┼───────────────────────────────────────────────┤
  │ safe      │ Read-only. Cannot modify files or run        │
  │           │ commands. Asks before everything.             │
  ├───────────┼───────────────────────────────────────────────┤
  │ standard  │ Can read files and run safe commands.         │
  │           │ Asks before writing files or running          │
  │           │ destructive commands. (DEFAULT)               │
  ├───────────┼───────────────────────────────────────────────┤
  │ power     │ Can read/write files and run most commands.   │
  │           │ Asks before git push, rm -rf, and network     │
  │           │ operations.                                   │
  ├───────────┼───────────────────────────────────────────────┤
  │ yolo      │ Full autonomy. No confirmations. Use only    │
  │           │ for throwaway/sandbox environments.           │
  └───────────┴───────────────────────────────────────────────┘

Switching Permission Modes #

text
> /permission safe

Permission mode: safe
  - File read: allowed
  - File write: BLOCKED (will suggest changes without applying)
  - Shell commands: BLOCKED
  - Git operations: BLOCKED

This mode is perfect for:
  - Learning and exploring
  - Reviewing code you don't want accidentally changed
  - Letting someone else use your machine with NeamCode
text
> /permission power

Permission mode: power
  - File read: auto-approved
  - File write: auto-approved
  - Shell commands (safe): auto-approved
  - Shell commands (destructive): ask first
  - Git operations: ask first
  - Network requests: ask first

This mode is perfect for:
  - Active development on your own projects
  - Refactoring sessions where you want speed
  - Building new features

Per-Project Permission Defaults #

Set permissions in your NEAMCODE.md:

markdown
## Permissions
- Default mode: power
- Never delete files without asking
- Never run npm publish
- Never push to main branch
⚠️ Warning

⚠️ Warning: Never use yolo mode on production systems, shared machines, or repositories with sensitive data. It is designed exclusively for throwaway environments like Docker containers or temporary sandbox projects.

24.8 CI/CD & Headless Mode #

NeamCode can run without a terminal UI, making it perfect for CI/CD pipelines, cron jobs, and automation scripts.

Basic Headless Usage #

bash
# Run a single prompt and get the result
neamcode --prompt "Review src/app.ts for security issues" --output text

# Get structured JSON output for parsing
neamcode --prompt "List all TODO comments in src/" --output json

# Set a cost limit for automated runs
neamcode --prompt "Analyze test coverage gaps" --output json --max-cost 0.50

CI/CD Integration Examples #

GitHub Actions — Automated PR Review:

yaml
name: NeamCode Review
on: [pull_request]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Install NeamCode
        run: npm install -g neamcode

      - name: Review PR
        run: |
          neamcode \
            --prompt "Review the git diff. Check for bugs, security issues, and style violations. Be concise." \
            --output json \
            --max-cost 0.25 \
            > review.json
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}

      - name: Post Review Comment
        run: |
          # Parse review.json and post as PR comment
          cat review.json | jq -r '.response' | gh pr comment ${{ github.event.number }} --body-file -
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

Pre-commit Hook:

bash
#!/bin/bash
# .git/hooks/pre-commit

# Quick check for common issues before committing
RESULT=$(neamcode \
  --prompt "Check staged files for console.log statements, hardcoded secrets, and TODO comments. Output only issues found." \
  --output text \
  --max-cost 0.05 \
  2>/dev/null)

if [ -n "$RESULT" ]; then
  echo "NeamCode found potential issues:"
  echo "$RESULT"
  echo ""
  echo "Commit anyway? (y/n)"
  read -r answer
  if [ "$answer" != "y" ]; then
    exit 1
  fi
fi

Cron Job — Nightly Code Quality Report:

bash
#!/bin/bash
# Run nightly at 2 AM

cd /var/app/my-project

neamcode \
  --prompt "Generate a code quality report. Check for: unused imports, functions over 50 lines, files over 300 lines, missing error handling, and missing tests. Format as markdown." \
  --output text \
  --max-cost 1.00 \
  > /var/reports/quality-$(date +%Y%m%d).md

# Email the report
mail -s "Daily Code Quality Report" team@company.com < /var/reports/quality-$(date +%Y%m%d).md

Headless Mode Options #

text
  ┌───────────────────┬────────────────────────────────────┐
  │ Flag              │ Description                        │
  ├───────────────────┼────────────────────────────────────┤
  │ --prompt "..."    │ The instruction to execute         │
  │ --output text     │ Plain text response                │
  │ --output json     │ Structured JSON response           │
  │ --output markdown │ Formatted markdown response        │
  │ --max-cost N.NN   │ Maximum dollar cost for this run   │
  │ --provider name   │ AI provider to use                 │
  │ --model name      │ Specific model to use              │
  │ --timeout N       │ Timeout in seconds                 │
  │ --context file    │ Additional context file to load    │
  │ --permission mode │ Permission level (safe recommended)│
  └───────────────────┴────────────────────────────────────┘
💡 Tip

💡 Tip: In CI/CD, always set --max-cost to prevent runaway spending. A typical code review costs $0.05-0.20. Set a generous but bounded limit like $0.50.

24.9 Security Best Practices #

NeamCode has access to your files and can run commands. Here is how to use it safely.

API Key Management #

text
# GOOD: Use environment variables
export OPENAI_API_KEY="sk-..."

# GOOD: Use a .env file (add to .gitignore!)
echo "OPENAI_API_KEY=sk-..." > .env
echo ".env" >> .gitignore

# BAD: Never hardcode keys in NEAMCODE.md or source code
# BAD: Never commit .env files to git

Permission Mode Selection Guide #

text
  ┌─────────────────────────────────┬──────────────┐
  │ Situation                       │ Mode         │
  ├─────────────────────────────────┼──────────────┤
  │ Exploring unfamiliar codebase   │ safe         │
  │ Client or shared project        │ standard     │
  │ Your own active project         │ power        │
  │ Learning / tutorials            │ safe         │
  │ CI/CD automation                │ safe or std  │
  │ Throwaway Docker container      │ yolo         │
  │ Production server               │ NEVER use    │
  └─────────────────────────────────┴──────────────┘

What NeamCode Should Never See #

Add these to your .gitignore and keep them out of NeamCode's reach:

text
# Secrets
.env
.env.local
.env.production
*.pem
*.key
credentials.json
service-account.json

# Sensitive data
*.sqlite (if it contains user data)
backups/
dumps/

Audit Trail #

NeamCode logs all file modifications and commands it runs. Review these periodically:

text
> /history

Recent actions this session:
  10:23  READ   src/app.ts
  10:24  READ   src/routes/users.ts
  10:25  WRITE  src/routes/users.ts (added input validation)
  10:26  RUN    npm test
  10:27  READ   tests/users.test.ts
  10:28  WRITE  tests/users.test.ts (added 3 test cases)
  10:29  RUN    npm test
📝 Note

📝 Note: NeamCode never sends your source code to external servers beyond what is needed for AI inference. Your code goes to the AI provider you selected and nowhere else. When using Ollama, your code never leaves your machine.

json
{
  "theme": "dracula",
  "vim_mode": false,
  "spinner": "dots",

  "auto_save_sessions": true,
  "compact_mode": false

Shell Aliases for Common Workflows #

Add these to your .zshrc or .bashrc:

bash
# Quick review current changes
alias ncreview='neamcode --prompt "/diff && /review --diff" --output text'

# Debug mode shortcut
alias ncdebug='neamcode --prompt "/mode debug"'

# Cost-free local mode
alias nclocal='neamcode --provider ollama --model deepseek-coder-v2'

# Explore a new codebase
alias ncexplore='neamcode --prompt "/mode explore" --permission safe'

Use Checkpoints Before Risky Changes #

text
> /checkpoint before-major-refactor

... do risky refactoring ...

> /restore before-major-refactor

text

# Switch to debug mode when you find an issue

> There's a race condition in the payment handler

> /mode test
> Write tests for the fixed payment handler
