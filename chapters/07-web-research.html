7.1 Web Search #

The web_search skill lets NeamCode search the internet for information — documentation, Stack Overflow answers, blog posts, changelogs — without you leaving the terminal.

Basic Web Search #

text
You: How do I set up ESLint with TypeScript in a Next.js project?

NeamCode:
  [web_search] "ESLint TypeScript Next.js setup 2026"

  Based on the current documentation and community best practices:

  1. Install the required packages:
     npm install -D eslint @typescript-eslint/parser
       @typescript-eslint/eslint-plugin eslint-config-next

  2. Next.js 14+ includes ESLint config out of the box.
     Your .eslintrc.json should extend:

     {
       "extends": [
         "next/core-web-vitals",
         "plugin:@typescript-eslint/recommended"
       ]
     }

  3. Add a lint script to package.json:
     "lint": "next lint"

  Source: Next.js documentation (nextjs.org/docs/app/building-
  your-application/configuring/eslint)

Searching for Error Solutions #

text
You: I'm getting "TypeError: Cannot read properties of undefined
     (reading 'map')" in my React component. Why?

NeamCode:
  [web_search] "TypeError Cannot read properties undefined map React"

  This is one of the most common React errors. It means you're
  calling .map() on a variable that is `undefined`. Common causes:

  1. **Data hasn't loaded yet.** If you're fetching data
     asynchronously, the initial render has `undefined`:

     // Problem:
     const [items, setItems] = useState();
     return items.map(...)  // 💥 items is undefined

     // Fix:
     const [items, setItems] = useState([]);
     return items.map(...)  // ✓ empty array, safe to map

  2. **API response shape changed.** You're accessing
     `response.data.items` but the API now returns
     `response.data.results`.

  3. **Optional chaining missing.** Use `items?.map(...)` for
     defensive coding.

  Want me to look at your component and find the specific issue?

Searching for API Documentation #

text
You: What are the Stripe API parameters for creating a PaymentIntent?

NeamCode:
  [web_search] "Stripe API create PaymentIntent parameters"

  Stripe PaymentIntent.create() accepts these key parameters:

  Required:
  - amount (integer) — Amount in smallest currency unit (cents)
  - currency (string) — Three-letter ISO currency code

  Common optional:
  - payment_method_types (array) — e.g., ['card']
  - customer (string) — Stripe Customer ID
  - description (string) — Internal description
  - metadata (object) — Key-value pairs for your use
  - confirm (boolean) — Confirm immediately if true
  - receipt_email (string) — Send receipt to this email

  Example:
    const intent = await stripe.paymentIntents.create({
      amount: 2000,      // $20.00
      currency: 'usd',
      payment_method_types: ['card'],
    });

  Source: Stripe API reference (stripe.com/docs/api/payment_intents)
📝 Note

📝 Note: NeamCode cites its sources when searching the web. This lets you verify the information if needed. Always check the source for critical implementation details — web search provides good starting points, but official docs are the final authority.

7.2 Fetching Web Pages #

The web_fetch skill retrieves and parses the content of a specific URL. Use this when you know exactly which page has the information you need.

Fetching Documentation #

text
You: Fetch the Prisma documentation page for migrations

NeamCode:
  [web_fetch] https://www.prisma.io/docs/concepts/components/
    prisma-migrate

  Here's a summary of the Prisma Migrate documentation:

  Prisma Migrate is a database migration tool. Key commands:

  • npx prisma migrate dev — Create and apply migrations
    (development)
  • npx prisma migrate deploy — Apply pending migrations
    (production)
  • npx prisma migrate reset — Reset database and reapply
    all migrations

  Workflow:
  1. Edit schema.prisma
  2. Run `prisma migrate dev --name description`
  3. Prisma generates SQL, creates migration file, applies it
  4. Commit the migration files to version control

  Important: Never edit migration files after they've been applied.

Reading PDFs from the Web (v0.3.5.3) #

NeamCode can now fetch and extract text from PDF files directly from URLs. This is especially useful for reading research papers, technical specifications, and documentation distributed as PDFs.

text
You: Read this paper: https://arxiv.org/pdf/2603.04448

NeamCode:
  [web_fetch] https://arxiv.org/pdf/2603.04448
    → Redirected to https://arxiv.org/abs/2603.04448
      (richer HTML abstract page)

  **Title:** Language Model Agents as Digital Scientists
  **Author:** Wang et al.
  **Pages:** 42

  The paper proposes a framework for using LLM-based agents
  as automated scientific researchers. Key contributions:

  1. A multi-agent architecture where agents collaborate on
     hypothesis generation, experiment design, and analysis
  2. Evaluation on 12 scientific benchmarks showing...
  (continues with extracted content)

How it works:

  • arxiv smart redirect — URLs like arxiv.org/pdf/XXXX are automatically redirected to the abstract page (arxiv.org/abs/XXXX), which has richer structured content
  • Binary PDF extraction — If the URL serves a raw PDF, NeamCode downloads the binary and extracts text using pdf-parse, supporting up to 200 pages
  • Metadata extraction — Title, author, and page count are automatically pulled from PDF metadata
  • Fallback — If the redirect fails, NeamCode falls back to extracting text from the raw PDF
text
You: Fetch this technical spec PDF and summarize the API changes
     https://example.com/api-spec-v3.pdf

NeamCode:
  [web_fetch] https://example.com/api-spec-v3.pdf

  **Pages:** 28

  API Specification v3.0 — Summary of Changes:

  1. Authentication: Migrated from API keys to OAuth 2.0
  2. Rate limiting: New tiered limits (100/1000/10000 req/min)
  3. Breaking changes:
     - POST /users now requires "email" field
     - GET /items response format changed to paginated
  ...
💡 Tip

💡 Tip: For academic research, just paste the arxiv PDF URL. NeamCode handles the redirect and extraction automatically — no need to manually download the PDF first.

7.3 Analyzing Screenshots #

The image_read skill lets NeamCode look at screenshots, mockups, and visual references. This bridges the gap between design and code.

Matching a Design Mockup #

text
You: Here's a screenshot of the design we need to implement.
     [provides mockup.png]

NeamCode:
  [image_read] mockup.png

  I can see a card-based product listing with:

  - Header bar with logo (left) and search + cart icon (right)
  - 3-column grid of product cards
  - Each card has:
    - Product image (square, rounded corners)
    - Product name (bold, ~16px)
    - Price in green ($XX.XX format)
    - "Add to Cart" button (blue, full-width)
  - Pagination dots at the bottom

  Want me to create the React component and CSS for this layout?

Debugging UI Issues #

text
You: The layout is broken on mobile. Here's a screenshot.
     [provides mobile-bug.png]

NeamCode:
  [image_read] mobile-bug.png

  I can see two issues:

  1. The sidebar navigation is overlapping the main content
     instead of collapsing into a hamburger menu.
  2. The product grid is still showing 3 columns instead
     of stacking into a single column.

  These are both CSS responsive breakpoint issues. Let me check
  your styles.

  [grep] "@media" in src/**/*.css

  Found the issue — your breakpoint is set at 480px but most
  mobile devices are 375-428px wide. Let me fix it.

7.4 Combining Research with Coding #

The real power emerges when you combine research skills with coding skills in a single workflow.

The Research-Implement-Verify Loop #

text
You: I need to add rate limiting to my Express API.
     Research the best approach and implement it.

NeamCode:
  [web_search] "Express rate limiting best practices 2026"

  The recommended approach is `express-rate-limit`. Let me
  implement it.

  [bash] npm install express-rate-limit (awaiting approval)
  ⟹ Allow? (y/n): y

  [file_read] src/app.ts

  [file_edit] src/app.ts (awaiting approval)

  old_string:
    import express from 'express';
    import cors from 'cors';

  new_string:
    import express from 'express';
    import cors from 'cors';
    import rateLimit from 'express-rate-limit';

  ⟹ Allow? (y/n): y

  [file_edit] src/app.ts (awaiting approval)

  old_string:
    app.use(cors());
    app.use(express.json());

  new_string:
    app.use(cors());
    app.use(express.json());

    const limiter = rateLimit({
      windowMs: 15 * 60 * 1000,  // 15 minutes
      max: 100,                    // limit each IP to 100 requests
      standardHeaders: true,
      legacyHeaders: false,
      message: { error: 'Too many requests, please try again later' }
    });
    app.use('/api', limiter);

  ⟹ Allow? (y/n): y

  [bash] npm test (awaiting approval)
  ⟹ Allow? (y/n): y

  ✓ All tests passing. Rate limiting is configured for 100
  requests per 15-minute window on all /api routes.

Debugging with Documentation #

text
You: My Prisma query is returning null when I know data exists.
     Here's my code:

     const user = await prisma.user.findFirst({
       where: { email: email }
     });

NeamCode:
  [web_search] "Prisma findFirst returning null case sensitivity"

  I found the issue. Prisma queries are **case-sensitive** by
  default with PostgreSQL. If the email in your database is
  "User@Example.com" but the query email is "user@example.com",
  it won't match.

  Fix:

  [file_edit] src/services/userService.ts (awaiting approval)

  old_string:
    const user = await prisma.user.findFirst({
      where: { email: email }
    });

  new_string:
    const user = await prisma.user.findFirst({
      where: {
        email: {
          equals: email,
          mode: 'insensitive'
        }
      }
    });

  ⟹ Allow? (y/n): y
  ✓ Fixed. Prisma will now do case-insensitive email matching.

     We're on 6.1.0 — check if we should upgrade.

NeamCode:
  [web_fetch] https://registry.npmjs.org/helmet/latest

  Current latest: helmet 8.0.0

  You're on 6.1.0, which is 2 major versions behind.

  [web_search] "helmet npm changelog breaking changes v7 v8"

  Breaking changes to be aware of:

  v7.0.0:
  - Dropped Node 14 support
  - crossOriginResourcePolicy enabled by default
  - contentSecurityPolicy default directives changed

  v8.0.0:
  - Dropped Node 16 support
  - Removed deprecated xssFilter (browsers handle this natively)
  - New strictTransportSecurity defaults

  thing to watch is the CSP default changes. Want me to upgrade
  and update your CSP config?

7.5 Pro Tips #

1. Be Specific with Search Queries #

text
# Good:
You: Search for how to implement WebSocket authentication in Express

# Less useful:
You: Search for WebSockets

NeamCode formulates good search queries, but specific questions get better results.

2. Fetch Official Docs Over Blog Posts #

text
You: Fetch the official Next.js docs on server actions,
     not blog posts

# Official docs are more accurate and up-to-date.

3. Use Image Analysis for UI Work #

text
You: Look at this Figma export and build the component

# Saves you from manually translating visual designs into code.
# NeamCode can see colors, spacing, layout, and typography.

4. Combine Multiple Research Sources #

text
You: I need to choose between Redis and Memcached for caching.
     Research both and give me a comparison for my use case
     (session storage for a Node.js app with 10k concurrent users).

NeamCode:
  [web_search] "Redis vs Memcached session storage Node.js"
  [web_search] "Redis session store performance 10k users"
  [web_fetch] https://redis.io/docs/latest/develop/use/
    client-side-caching/

  (Detailed comparison with recommendation)

5. Let Research Flow Into Implementation #

Don't separate research and coding into different sessions. The best workflow is:

│        │                                         │
│        ▼                                         │

│   │ web_search│   │  Results  │                  │
│   └──────────┘    └─────┬─────┘                  │

│                   ┌───────────┐                   │
│                   │ Implement │                   │

│                   └─────┬─────┘                  │
│                         │                        │

All of this happens in a single NeamCode conversation. No tab-switching, no copy-pasting from browser to terminal.

Exercises #

🎯 Exercise 7.2: Pick an npm package (or pip/gem/cargo package) you depend on. Ask NeamCode to fetch its changelog and tell you if there are any breaking changes between your version and the latest.

🎯 Exercise 7.3: Take a screenshot of any web page. Ask NeamCode to analyze it and describe the layout structure. Then ask it to recreate the layout in HTML and CSS.
