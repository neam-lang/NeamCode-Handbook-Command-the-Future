21.1 Code Review Workflows #

Code review is where bugs are caught and knowledge is shared. NeamCode's /review command gives you a thorough first pass before human reviewers see the code.

Pre-Review Your Own Code #

text
$ cd ~/work/payment-service
$ neamcode

> /review src/handlers/checkout.ts

Reviewing src/handlers/checkout.ts...

## Code Review: checkout.ts

### Critical Issues (2)

🔴 **SQL Injection vulnerability** (line 47)

const query = SELECT * FROM users WHERE id = '${userId}';

text
   Use parameterized queries instead:

const query = 'SELECT * FROM users WHERE id = $1'; const result = await db.query(query, [userId]);

text

🔴 **Race condition in inventory check** (lines 62-68)
   The stock check and decrement are not atomic:

const stock = await getStock(itemId); // Read if (stock > 0) { await decrementStock(itemId); // Write

text
   Between the read and write, another request could claim the
   same item. Use a database transaction with SELECT FOR UPDATE.

### Warnings (3)

🟡 **No input validation** (line 23)
   `req.body.amount` is used directly without validation.
   Add: `z.number().positive().max(999999)`

🟡 **Missing error handling** (line 71)
   `await chargeCard(...)` can throw but isn't wrapped in try/catch.
   Payment failures would crash the handler.

🟡 **Hardcoded timeout** (line 89)
   `timeout: 5000` should be configurable via environment variable.

### Suggestions (2)

💬 Extract lines 30-55 into a separate `validateOrder()` function.
💬 Add structured logging with request ID for traceability.

Review a Diff Before Committing #

text
> /diff

Showing staged changes...

Modified: src/handlers/checkout.ts (+42, -15)
Modified: src/models/order.ts (+8, -2)
Added: src/validators/checkout.validator.ts (+35)

> /review --diff

Reviewing staged changes only...

## Diff Review

✅ SQL injection fix looks correct — parameterized queries used.
✅ New validator file properly validates amount, currency, items.
✅ Order model changes are backward-compatible.

⚠️  The race condition fix uses a transaction but doesn't set
    isolation level. Add:
    await db.query('SET TRANSACTION ISOLATION LEVEL SERIALIZABLE');

⚠️  checkout.validator.ts exports `validateCheckout` but the import
    in checkout.ts uses `validateCheckoutRequest` (name mismatch).
    This will fail at runtime.

Otherwise, good changes. Ready to commit after fixing the import.

Review a Pull Request #

text
> Review this PR: git_pr https://github.com/ourteam/api/pull/342

Fetching PR #342: "Add rate limiting to public endpoints"...

## PR Review: #342

**Summary:** Adds Redis-based rate limiting middleware to all
public API endpoints. 150 requests/minute per IP.

**Files changed:** 6 files, +187, -12

### Architecture: ✅ Good
  - Middleware pattern is correct for Express
  - Redis is the right choice for distributed rate limiting
  - Sliding window algorithm prevents burst abuse

### Issues Found:

🔴 **IPv6 handling** (middleware/rateLimit.ts:18)
   `req.ip` can be IPv6 on dual-stack servers. Your Redis key
   `rate:${req.ip}` would create separate limits for IPv4 and
   IPv6 from the same client. Normalize to IPv4 when possible.

🟡 **No bypass for health checks** (middleware/rateLimit.ts:5)
   Load balancer health checks will consume rate limit quota.
   Add: `if (req.path === '/health') return next();`

🟡 **Redis connection failure** (middleware/rateLimit.ts:31)
   If Redis is down, the middleware throws and blocks ALL requests.
   Add a fallback: if Redis unavailable, allow requests through
   (fail-open) or use in-memory fallback.

💬 Consider adding `Retry-After` header to 429 responses.
💬 Add rate limit info headers: `X-RateLimit-Remaining`, etc.
💡 Tip

💡 Tip: Run /review on your own code before requesting human review. It catches the obvious issues so your reviewers can focus on architecture and logic.

21.2 Microservice Architecture #

NeamCode helps you design, scaffold, and refactor microservice architectures.

Designing a New Service #

text
> /mode analyze

> We need to extract the notification logic from our monolith into
  a standalone service. Currently notifications are sent inline
  during order processing. We need email, SMS, push, and Slack.
  Design the service architecture.

## Notification Service Architecture

    ┌──────────────┐
    │ Order Service │
    │  (existing)   │─────────┐
    └──────────────┘          │
                              ▼
    ┌──────────────┐    ┌───────────┐    ┌──────────────────┐
    │ User Service  │───▶│  Message   │───▶│ Notification     │
    │  (existing)   │    │  Queue     │    │ Service (new)    │
    └──────────────┘    │ (RabbitMQ) │    │                  │
                        └───────────┘    │ ┌──────────────┐ │
                                         │ │ Router       │ │
                                         │ │ (type→chan.) │ │
                                         │ └──────┬───────┘ │
                                         │   ┌────┼────┐    │
                                         │   ▼    ▼    ▼    │
                                         │ Email SMS  Push  │
                                         │ (SES)(Twl)(FCM)  │
                                         └──────────────────┘

**Key decisions:**

1. **Async via message queue** — Order service publishes events,
   doesn't wait for notification delivery. Decouples latency.

2. **Channel abstraction** — Each channel (email, SMS, push, Slack)
   implements a common `NotificationChannel` interface:

interface NotificationChannel { send(recipient: string, message: Message): Promise<Result>; checkStatus(id: string): Promise<DeliveryStatus>; }

text

3. **Event schema:**

{ "event": "order.completed", "user_id": "usr_abc", "channels": ["email", "push"], "template": "order_confirmation", "data": { "order_id": "ord_123", "total": 49.99 } }

text

4. **Retry with exponential backoff** — Failed sends retry 3 times
   with delays of 1s, 5s, 30s. Dead letter queue after that.

5. **Template engine** — Notifications use Handlebars templates
   stored in the database, not hardcoded strings.

21.3 Database Migrations #

Database migrations are high-risk operations. NeamCode helps you write safe, reversible migrations.

Generating a Migration #

text
> I need to add a 'preferences' JSONB column to the users table
  and backfill it with default values. We have 12 million rows.
  PostgreSQL 15. Generate a safe migration.

## Migration: add_user_preferences

Created: migrations/20260307_add_user_preferences.sql

**UP migration:**

  -- Step 1: Add column with default (instant, no table rewrite)
  ALTER TABLE users
    ADD COLUMN preferences JSONB
    DEFAULT '{"email_notifications": true, "theme": "system"}'::jsonb;

  -- Step 2: Create index concurrently (non-blocking)
  CREATE INDEX CONCURRENTLY idx_users_preferences_email
    ON users USING gin ((preferences->'email_notifications'));

  -- Step 3: Backfill in batches (12M rows, 10K per batch)
  DO $$
  DECLARE
    batch_size INT := 10000;
    total_updated INT := 0;
  BEGIN
    LOOP
      UPDATE users
      SET preferences = '{"email_notifications": true, "theme": "system"}'::jsonb
      WHERE preferences IS NULL
        AND id IN (
          SELECT id FROM users WHERE preferences IS NULL LIMIT batch_size
        );
      GET DIAGNOSTICS total_updated = ROW_COUNT;
      EXIT WHEN total_updated = 0;
      RAISE NOTICE 'Updated % rows so far', total_updated;
      PERFORM pg_sleep(0.1);  -- Brief pause to reduce lock pressure
    END LOOP;
  END $$;

**DOWN migration:**

  ALTER TABLE users DROP COLUMN preferences;

⚠️ **Safety notes:**
  - Adding a column with DEFAULT is instant in PostgreSQL 11+
    (no table rewrite)
  - CREATE INDEX CONCURRENTLY doesn't lock the table for writes
  - Backfill runs in 10K batches to avoid long-running transactions
  - Estimated backfill time for 12M rows: ~5 minutes
  - Test on staging with production-like data volume first
⚠️ Warning

⚠️ Warning: Always run migrations on staging first. Always have a tested rollback plan. Always take a backup before migrating production.

21.4 Performance Optimization #

When your service is slow, NeamCode helps you find the bottleneck and fix it systematically.

Profiling and Analysis #

text
> /mode debug

> Our API endpoint GET /api/orders takes 3.2 seconds on average.
  It should be under 200ms. Here's the handler code and the SQL
  query it runs. Help me find the bottleneck.

Let me analyze the code path...

## Performance Analysis: GET /api/orders

**Bottleneck identified: N+1 query problem**

Your handler does this:
  1. Fetch orders: SELECT * FROM orders WHERE user_id = $1    (12ms)
  2. For EACH order (avg 47 orders per user):
     - Fetch items: SELECT * FROM order_items WHERE order_id = $1  (15ms × 47)
     - Fetch product: SELECT * FROM products WHERE id = $1         (8ms × 47)

  Total queries: 1 + (47 × 2) = 95 queries
  Total DB time: 12 + (47 × 23) = 1,093ms
  Network overhead: ~95 × 22ms roundtrip = 2,090ms

  Total: ~3,183ms  ← matches your 3.2s observation

**Fix: Single query with JOINs**

  SELECT
    o.id, o.created_at, o.status, o.total,
    oi.quantity, oi.price,
    p.name, p.image_url
  FROM orders o
  JOIN order_items oi ON oi.order_id = o.id
  JOIN products p ON p.id = oi.product_id
  WHERE o.user_id = $1
  ORDER BY o.created_at DESC;

  This reduces 95 queries to 1 query.

  Additional optimizations:
  1. Add composite index:
     CREATE INDEX idx_orders_user_created
       ON orders(user_id, created_at DESC);

  2. Add pagination (don't return all 47 orders at once):
     LIMIT 20 OFFSET $2

  3. Add Redis cache with 60s TTL for repeat requests.

**Expected result after fix:**
  1 query: ~25ms
  Serialization: ~5ms
  Total: ~30ms (100x improvement)

Using NeamCode in CI/CD #

text
> I want to run NeamCode as part of our CI to auto-review PRs.
  How do I set that up?

Add this job to your workflow:

  neamcode-review:
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: NeamCode PR Review
        run: |
          neamcode \
            --prompt "Review the diff in this PR. Focus on security, \
                      performance, and correctness. Output as a \
                      structured review with severity levels." \
            --output json \
            --max-cost 0.50
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}

This runs NeamCode in headless mode. It reads the diff, reviews
the code, and outputs structured JSON. You can pipe that output
to a script that posts it as a PR comment.

Cost per review: typically $0.05-0.15 depending on diff size.
💡 Tip

💡 Tip: Set --max-cost in CI to prevent runaway costs. A typical code review costs under $0.20. Set the limit to $0.50 for safety.

21.6 Incident Response #

When production is on fire, NeamCode helps you diagnose and fix issues fast.

Rapid Diagnosis #

text
$ neamcode

> /mode debug

> URGENT: Our order-service is returning 500 errors for 30% of
  requests. Started 15 minutes ago. No deploys in the last 2 hours.
  Here's the error from our logs:

  Error: ECONNREFUSED 10.0.3.42:5432
  at TCPConnectWrap.afterConnect

Let me help you triage this quickly.

## Incident Analysis

**Root cause (high confidence): Database connection failure**

The error `ECONNREFUSED 10.0.3.42:5432` means:
  - Your app cannot connect to PostgreSQL at 10.0.3.42
  - Port 5432 is the default PostgreSQL port
  - 30% failure rate suggests: connection pool exhaustion
    or intermittent network issue (not full outage)

**Immediate triage steps:**

  1. Check if PostgreSQL is running:
     $ ssh db-server 'pg_isready -h 10.0.3.42'

  2. Check connection pool status:
     $ curl localhost:3000/health | jq '.database'

  3. Check active connections:
     $ psql -h 10.0.3.42 -c \
       "SELECT count(*) FROM pg_stat_activity;"

  4. Check if connection limit is hit:
     $ psql -h 10.0.3.42 -c \
       "SHOW max_connections;"

**Most likely scenarios:**
  A. Connection pool exhausted → restart pods or increase pool
  B. Database hit max_connections → kill idle connections
  C. Network partition → check security groups / NACLs
  D. Disk full on DB server → check df -h on db-server

Which of these checks can you run now? Or paste the output
and I'll analyze further.

Writing the Postmortem #

text
> Incident is resolved. It was connection pool exhaustion — a
  background job was leaking connections. Help me write the
  postmortem.

## Postmortem: Order Service 500 Errors

**Date:** 2026-03-07
**Duration:** 47 minutes (14:23 - 15:10 UTC)
**Severity:** SEV-2
**Impact:** 30% of order API requests returned 500 errors

### Timeline
  14:23  Monitoring alert: error rate > 5%
  14:25  On-call engineer paged
  14:31  Identified ECONNREFUSED errors in logs
  14:38  Confirmed PostgreSQL connection pool exhausted
  14:42  Identified leaking background job (invoice-generator)
  14:45  Killed invoice-generator pods
  14:47  Connection pool recovered, error rate dropping
  15:10  Error rate at 0%, incident resolved

### Root Cause
The invoice-generator background job (deployed 3 days ago)
opened a new database connection for each invoice but never
released it. At ~200 invoices/hour, it exhausted the connection
pool limit (100 connections) after ~72 hours.

### Resolution
Killed invoice-generator pods to release connections.
Hotfix deployed: added `finally { connection.release() }` to
the invoice generation function.

### Action Items
  - [ ] Add connection pool monitoring with alert at 80% (P1)
  - [ ] Add connection leak detection to CI tests (P2)
  - [ ] Increase connection pool from 100 to 200 (P3)
  - [ ] Add connection timeout to prevent indefinite hangs (P2)

### Lessons Learned
  - Background jobs need the same code review rigor as API handlers
  - Connection pool metrics should have been in our dashboard
  - The 72-hour time bomb pattern is hard to catch without
    load testing at scale

21.7 Legacy Code Modernization #

NeamCode helps you understand and incrementally modernize legacy systems.

Understanding Legacy Code #

text
> /mode analyze

> We have a 15-year-old PHP monolith. I need to understand what
  this 800-line function does before refactoring it. The original
  developer left 10 years ago. No documentation.

  [pastes processOrder() function]

## Analysis of processOrder() — 800 lines

This function does **seven** distinct things (violation of Single
Responsibility Principle):

  processOrder()
  ├── Lines 1-85:     Validate order inputs
  ├── Lines 86-210:   Check inventory across 3 warehouses
  ├── Lines 211-340:  Calculate pricing (discounts, tax, shipping)
  ├── Lines 341-445:  Process payment (Stripe + legacy PayPal)
  ├── Lines 446-560:  Update inventory and create shipment
  ├── Lines 561-680:  Send notifications (email + legacy fax!)
  └── Lines 681-800:  Logging, analytics, loyalty points

**Hidden behaviors I found:**

  ⚠️  Line 127: If warehouse_id = 3, adds a 2% "handling fee"
     (looks like a hack from 2014, comment says "temporary")

  ⚠️  Line 389: Catches ALL exceptions and retries payment 5 times
     with no backoff. Could charge customer 5 times if partial
     success occurs.

  ⚠️  Line 590: Sends fax notification if customer.fax_number is set.
     Fax library hasn't been updated since PHP 5.3.

  ⚠️  Line 712: Writes directly to a CSV file for "analytics".
     File path is hardcoded to /var/www/orders.csv.

**Refactoring plan (incremental, safe):**

  Phase 1: Extract validation → OrderValidator class
  Phase 2: Extract pricing → PricingCalculator class
  Phase 3: Extract payment → PaymentProcessor class
  Phase 4: Extract notifications → NotificationService
  Phase 5: Extract inventory → InventoryService
  Phase 6: Remove fax support (confirm with stakeholders first)
  Phase 7: Replace CSV analytics with proper event tracking

Each phase can be deployed independently. Start with Phase 1.

21.10 Pro Tips for Professionals #

Use Permission Modes Strategically #

text
> /permission power

Permission mode: power
  - File read/write: auto-approved
  - Shell commands: auto-approved for safe commands
  - Git operations: ask for confirmation
  - Network requests: ask for confirmation

For trusted projects you own, power mode saves confirmation clicks. For client code or production systems, stay on standard.

Team-Wide NEAMCODE.md #

Put a NEAMCODE.md in your project root:

markdown
# Team Conventions

## Stack
- TypeScript 5.3, Node.js 20 LTS
- PostgreSQL 15, Redis 7
- Express 4.x (migrating to Fastify)
- Jest for testing, 80% coverage required

## Code Style
- Strict TypeScript (no any)
- Functional style preferred, classes for services
- All functions must have JSDoc comments
- Error handling: use Result<T, E> pattern, not try/catch

## PR Requirements
- All PRs need at least 1 approval
- Squash merge only
- PR title format: JIRA-123: description
- Must include tests for new functionality

Every team member using NeamCode gets consistent suggestions that match your standards.

Headless Mode for Automation #

bash
# Automated nightly code quality report
neamcode \
  --prompt "Analyze src/ for code smells, unused imports, \
            and potential memory leaks. Output a summary." \
  --output json \
  --max-cost 1.00 \
  > reports/quality-$(date +%Y%m%d).json

# Automated PR review bot
neamcode \
  --prompt "Review the git diff for this PR. Check for security \
            issues, performance problems, and style violations. \
            Reference our NEAMCODE.md conventions." \
  --output json \
  --max-cost 0.50
