6.1 Running Shell Commands (bash) #

The bash skill lets NeamCode execute any shell command on your behalf. This is one of its most powerful — and most carefully guarded — capabilities.

text

NeamCode:

  v20.11.0

6.2 Background Tasks (background_shell) #

Some commands take a long time — dev servers, watch modes, long builds. The background_shell skill runs these without blocking your conversation.

Starting a Dev Server #

text
You: Start the dev server in the background

NeamCode:
  [background_shell] npm run dev (awaiting approval)
  ⟹ Allow? (y/n): y

  ✓ Started in background (PID: 42381)

  Server is starting up. You can continue working while it runs.
  The dev server will be available at http://localhost:3000.

Running Watch Mode #

text
You: Start the test watcher in the background

NeamCode:
  [background_shell] npm test -- --watch (awaiting approval)
  ⟹ Allow? (y/n): y

  ✓ Started in background (PID: 42399)

  Tests are running in watch mode. They'll re-run automatically
  when files change. You can keep coding and I'll keep editing —
  the watcher runs independently.

Managing Background Tasks #

text
You: What's running in the background?

NeamCode: You have 2 background processes:

  PID 42381 — npm run dev (running, port 3000)
  PID 42399 — npm test --watch (running)

You: Stop the test watcher

NeamCode:
  [bash] kill 42399 (awaiting approval)
  ⟹ Allow? (y/n): y

  ✓ Test watcher stopped.
💡 Tip

💡 Tip: Use background_shell for anything that runs indefinitely: dev servers, file watchers, database instances, log tailing. Use regular bash for commands that complete and return output.

bash
  • Runs command, waits for it to finish
  • Returns the output
  • Good for: builds, tests, scripts, git
background_shell
  • Starts command and returns immediately
  • Process runs in the background
  • Good for: dev servers, watchers, long tasks

text

NeamCode:

Smart Commit Messages #

text
You: Commit these changes with an appropriate message

NeamCode:
  [bash] git diff --staged

  Based on the staged changes, here's my suggested commit:

  [bash] git commit -m "fix: Resolve race condition in checkout stock validation

  ⟹ Allow? (y/n): y

💡 Tip

💡 Tip: NeamCode reads your diff to generate meaningful commit messages. It follows conventional commit format (feat:, fix:, refactor:, etc.) automatically.

Handling Merge Conflicts #

text
You: Merge feature/payment-integration

NeamCode:
  [bash] git merge feature/payment-integration (awaiting approval)
  ⟹ Allow? (y/n): y

  Auto-merging src/routes/index.ts
  CONFLICT (content): Merge conflict in src/routes/index.ts

  There's a merge conflict in src/routes/index.ts. Let me look at it.

  [file_read] src/routes/index.ts

  The conflict is in the route registration section:

  <<<<<<< HEAD
  router.use('/auth', authRoutes);
  =======
  router.use('/payments', paymentRoutes);
  >>>>>>> feature/payment-integration

  Both branches added a new route in the same location. The fix is
  simple — we need both routes.

  [file_edit] src/routes/index.ts (awaiting approval)

  old_string:
    <<<<<<< HEAD
    router.use('/auth', authRoutes);
    =======
    router.use('/payments', paymentRoutes);
    >>>>>>> feature/payment-integration

  new_string:
    router.use('/auth', authRoutes);
    router.use('/payments', paymentRoutes);

  ⟹ Allow? (y/n): y
  ✓ Conflict resolved

  [bash] git add src/routes/index.ts && git commit -m "merge: Resolve conflict in route registration" (awaiting approval)
  ⟹ Allow? (y/n): y
  ✓ Merge complete

6.7 The Permission System in Practice #

Understanding the permission system is essential when running commands and Git operations. Here's how it works in practice:

AUTO-APPROVED (no prompt)
  • file_read — Reading files is always safe
  • glob — Searching filenames is safe
  • grep — Searching file contents is safe
  • dir_tree — Viewing structure is safe
  • code_structure — Analyzing code is safe
REQUIRES APPROVAL (you see a prompt)
  • file_write — Creating/overwriting files
  • file_edit — Modifying existing files
  • bash — Running ANY shell command
  • background_shell — Starting background processes
  • git_pr — Creating pull requests
📝 Note

The principle: READ freely, WRITE with permission.

What Approval Looks Like #

Every time NeamCode wants to execute something that could change your system, you'll see:

text
  [bash] npm install express cors helmet (awaiting approval)

  ⟹ Allow this command? (y/n): y

You have three options:

  • y — approve and execute
  • n — reject the command (NeamCode will ask what you'd prefer instead)
  • Read the command first — take your time, there's no rush

Rejecting Commands #

text
NeamCode:
  [bash] rm -rf node_modules && npm install (awaiting approval)
  ⟹ Allow? (y/n): n

  Understood. Would you prefer I just run `npm install` without
  deleting node_modules first? That's usually sufficient unless
  you're having dependency resolution issues.
⚠️ Warning

⚠️ Warning: Always read the command before approving. NeamCode is careful, but you are the final authority. If a command looks wrong or too destructive, reject it and explain what you actually want.

4. Chain Analysis with Execution #

text
You: Run the tests, and if any fail, read the failing test
     file and the source file it's testing, then fix the issue.

# NeamCode will:
# 1. Run tests (bash)
# 2. Parse the output for failures
# 3. Read relevant files (file_read)
# 4. Edit the fix (file_edit)
# 5. Re-run tests to verify (bash)

5. Use /review Before Every PR #

Make it a habit:

text

You: Fix the issues found
You: /review
You: Looks good. Create the PR.

This review-fix-review cycle catches issues before your teammates see them.

6. Understand What Commands Do Before Approving #

If you're not sure what a command does:

text
NeamCode:
  [bash] find . -name "*.pyc" -delete (awaiting approval)

You: What does this command do exactly?

NeamCode: This command finds all files ending in .pyc (Python
  bytecode cache files) in the current directory and all
  subdirectories, then deletes them. These are safe to delete —
  Python regenerates them automatically. Want me to proceed?

  ⟹ Allow? (y/n): y
