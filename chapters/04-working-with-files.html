Chapter 4: Working with Files #

"The file system is a programmer's canvas. With NeamCode, you paint with words."
— The NeamCode Handbook

📖 Estimated read time: 20 minutes

What You'll Learn #

  • How to read files of any type with file_read
  • How to create new files with file_write
  • The precise old_string→new_string editing model that makes file_edit reliable
  • How to scaffold an entire project from scratch using natural language
  • Strategies for working with large files
  • Handling binary files and images

4.1 Reading Files #

Reading files is the most common operation you'll perform in NeamCode. The file_read skill loads file contents so NeamCode can understand, analyze, and discuss your code with you.

Basic File Reading #

Simply ask NeamCode to look at a file:

text
You: Show me the contents of server.js

NeamCode: I'll read that file for you.

  [file_read] server.js

const express = require('express');
const app = express();
const PORT = process.env.PORT || 3000;

app.get('/', (req, res) => {
  res.json({ message: 'Hello World' });
});

app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});

This is a basic Express.js server that:
- Listens on port 3000 (or a custom PORT from environment variables)
- Has a single GET route at "/" that returns a JSON message

Reading Multiple Files #

You can ask about several files at once:

text
You: Read package.json and server.js — do the dependencies match what's used?

NeamCode:
  [file_read] package.json
  [file_read] server.js

Looking at both files, I notice a mismatch. Your server.js imports
`express` and `cors`, but package.json only lists `express` as a
dependency. You're missing `cors`. Run:

  npm install cors

Partial File Reading #

For large files, you can ask for specific parts:

text
You: Show me just the first 50 lines of database.js

NeamCode:
  [file_read] database.js (lines 1-50)

  // ... first 50 lines displayed ...
💡 Tip

💡 Tip: File reads are auto-approved — they never require permission confirmation. NeamCode treats reading as a safe, non-destructive operation.

Reading with Context #

NeamCode doesn't just dump file contents — it understands them:

text
You: Read the User model and tell me if there are any security issues

NeamCode:
  [file_read] models/User.js

I found two security concerns:

1. **Password stored in plain text** (line 12): The password field has
   no hashing. Use bcrypt before saving.
2. **No input validation** (line 8): The email field accepts any string.
   Add a validation regex or use a library like `validator`.

4.2 Creating New Files #

When you need a new file, use natural language to describe what you want. NeamCode uses file_write to create it.

Simple File Creation #

text
You: Create a .gitignore for a Node.js project

NeamCode: I'll create a .gitignore file with common Node.js exclusions.

  [file_write] .gitignore (awaiting approval)

  ⟹ Allow this write? (y/n): y

  ✓ Created .gitignore

Contents written:
  node_modules/
  .env
  .env.local
  dist/
  build/
  coverage/
  *.log
  .DS_Store
  .idea/
  .vscode/
⚠️ Warning

⚠️ Warning: File writes always require your approval. NeamCode will show you what it's about to write and wait for confirmation. This is by design — you stay in control.

Creating Files with Specific Content #

text
You: Create a TypeScript interface file at src/types/user.ts with fields
     for id (number), name (string), email (string), and optional avatar (string)

NeamCode:
  [file_write] src/types/user.ts (awaiting approval)

  export interface User {
    id: number;
    name: string;
    email: string;
    avatar?: string;
  }

  ⟹ Allow this write? (y/n): y

  ✓ Created src/types/user.ts

Creating Multiple Files #

text
You: Create a basic React component called Header with its CSS module

NeamCode:
  [file_write] src/components/Header/Header.tsx (awaiting approval)
  ⟹ Allow this write? (y/n): y
  ✓ Created Header.tsx

  [file_write] src/components/Header/Header.module.css (awaiting approval)
  ⟹ Allow this write? (y/n): y
  ✓ Created Header.module.css

  [file_write] src/components/Header/index.ts (awaiting approval)
  ⟹ Allow this write? (y/n): y
  ✓ Created index.ts
💡 Tip

💡 Tip: If you trust NeamCode for a session, you can pre-approve certain write patterns. But for learning, keeping approvals on helps you understand exactly what's happening.

4.3 Editing Existing Code #

This is where NeamCode's approach really shines — and where understanding the underlying model pays off.

The old_string → new_string Model #

NeamCode's file_edit skill works by precise text matching. It finds an exact string in your file and replaces it with a new string. This is fundamentally different from line-number-based editing.

How file_edit Works
  1. NeamCode reads the current file
  2. It identifies the EXACT text to change
  3. It specifies:
    • old_string: the text to find
    • new_string: the text to replace it with
  4. The match must be UNIQUE in the file
  5. You approve → change applied
old_string
(must exist exactly once) → new_string
(replaces in place)

Basic Edit Example #

text
You: Change the port from 3000 to 8080 in server.js

NeamCode:
  [file_edit] server.js (awaiting approval)

  old_string:
    const PORT = process.env.PORT || 3000;

  new_string:
    const PORT = process.env.PORT || 8080;

  ⟹ Allow this edit? (y/n): y

  ✓ Edited server.js

Why old_string → new_string Matters #

This approach is more reliable than line-number editing because:

  1. No off-by-one errors. Line numbers shift when you add or remove lines. Text matching doesn't care about line numbers.
  2. Context-aware. The matched string provides surrounding context, so you can verify the right code is being changed.
  3. Transparent. You see exactly what's being removed and what's replacing it.
📝 Note

📝 Note: If the old_string appears more than once in the file, NeamCode will include enough surrounding context to make the match unique. If it can't find the exact text, the edit fails safely — nothing is changed.

Multi-Part Edits #

Sometimes a single change requires editing multiple locations:

text
You: Add CORS support to server.js

NeamCode:
  [file_read] server.js

  I'll make two edits: add the import and add the middleware.

  [file_edit] server.js (1/2) (awaiting approval)

  old_string:
    const express = require('express');

  new_string:
    const express = require('express');
    const cors = require('cors');

  ⟹ Allow this edit? (y/n): y
  ✓ Edited server.js

  [file_edit] server.js (2/2) (awaiting approval)

  old_string:
    const app = express();

  new_string:
    const app = express();
    app.use(cors());

  ⟹ Allow this edit? (y/n): y
  ✓ Edited server.js

When Edits Fail #

text
You: Change the database URL to localhost

NeamCode:
  [file_edit] config.js

  ✗ Edit failed: old_string not found in file.

  The file may have been modified since I last read it.
  Let me re-read and try again.

  [file_read] config.js

  I see — the database URL is in config/database.js, not config.js.
  Let me edit the correct file.

  [file_edit] config/database.js (awaiting approval)
  ...
💡 Tip

💡 Tip: If an edit fails, NeamCode will typically re-read the file and retry. This self-correction is a strength of the text-matching approach — failures are obvious and recoverable.

4.4 Building a Project from Scratch #

Let's put it all together. We'll ask NeamCode to scaffold a complete Node.js Express application with routes, middleware, and tests.

Step 1: Describe What You Want #

text
You: Create a Node.js Express REST API for a todo app.
     I want:
     - Express with JSON body parsing
     - Routes for CRUD on todos (in-memory array is fine)
     - Error handling middleware
     - A health check endpoint
     - Basic tests with Jest
     - package.json with scripts for start, dev, and test

Step 2: Watch NeamCode Work #

text
NeamCode: I'll create the full project structure. Here's my plan:

  todo-api/
  ├── package.json
  ├── .gitignore
  ├── src/
  │   ├── index.js          # Entry point
  │   ├── app.js             # Express app setup
  │   ├── routes/
  │   │   └── todos.js       # CRUD routes
  │   └── middleware/
  │       └── errorHandler.js
  └── tests/
      └── todos.test.js

  Let me create each file.

  [file_write] todo-api/package.json (awaiting approval)
  ⟹ Allow? (y/n): y ✓

  [file_write] todo-api/.gitignore (awaiting approval)
  ⟹ Allow? (y/n): y ✓

  [file_write] todo-api/src/app.js (awaiting approval)
  ⟹ Allow? (y/n): y ✓

  [file_write] todo-api/src/routes/todos.js (awaiting approval)
  ⟹ Allow? (y/n): y ✓

  [file_write] todo-api/src/middleware/errorHandler.js (awaiting approval)
  ⟹ Allow? (y/n): y ✓

  [file_write] todo-api/src/index.js (awaiting approval)
  ⟹ Allow? (y/n): y ✓

  [file_write] todo-api/tests/todos.test.js (awaiting approval)
  ⟹ Allow? (y/n): y ✓

  All files created! To get started:

    cd todo-api
    npm install
    npm run dev

Step 3: Iterate #

text
You: Add input validation — todos should have a title (required)
     and a completed field (boolean, defaults to false)

NeamCode:
  [file_read] todo-api/src/routes/todos.js

  [file_edit] todo-api/src/routes/todos.js (awaiting approval)

  old_string:
    router.post('/', (req, res) => {
      const todo = {
        id: nextId++,
        ...req.body
      };

  new_string:
    router.post('/', (req, res) => {
      const { title, completed } = req.body;

      if (!title || typeof title !== 'string' || title.trim() === '') {
        return res.status(400).json({ error: 'Title is required' });
      }

      const todo = {
        id: nextId++,
        title: title.trim(),
        completed: typeof completed === 'boolean' ? completed : false
      };

  ⟹ Allow? (y/n): y
  ✓ Edited todos.js

This iterative flow — scaffold, review, refine — is how experienced NeamCode users build real projects.

4.5 Working with Large Files #

Large files (hundreds or thousands of lines) require some strategy.

Reading Specific Sections #

text
You: Show me the handleSubmit function in src/components/Form.tsx

NeamCode:
  [file_read] src/components/Form.tsx

  Here's the handleSubmit function (lines 142-189):

  const handleSubmit = async (e: FormEvent) => {
    e.preventDefault();
    setLoading(true);
    // ... full function shown ...
  };

NeamCode is smart enough to read the whole file internally but show you only the relevant portion.

Targeted Edits in Large Files #

text
You: In the 800-line utils.js file, find the formatDate function
     and make it support ISO 8601 format

NeamCode:
  [file_read] src/utils.js

  Found formatDate at line 347. Currently it only supports
  'MM/DD/YYYY'. I'll add ISO 8601 support.

  [file_edit] src/utils.js (awaiting approval)

  old_string:
    function formatDate(date) {
      const mm = String(date.getMonth() + 1).padStart(2, '0');
      const dd = String(date.getDate()).padStart(2, '0');
      const yyyy = date.getFullYear();
      return `${mm}/${dd}/${yyyy}`;
    }

  new_string:
    function formatDate(date, format = 'MM/DD/YYYY') {
      const mm = String(date.getMonth() + 1).padStart(2, '0');
      const dd = String(date.getDate()).padStart(2, '0');
      const yyyy = date.getFullYear();

      switch (format) {
        case 'ISO':
          return date.toISOString();
        case 'YYYY-MM-DD':
          return `${yyyy}-${mm}-${dd}`;
        case 'MM/DD/YYYY':
        default:
          return `${mm}/${dd}/${yyyy}`;
      }
    }

  ⟹ Allow? (y/n): y
  ✓ Edited src/utils.js
💡 Tip

💡 Tip: When working with large files, be specific about what you want changed. Instead of "refactor utils.js," say "refactor the formatDate function in utils.js to support multiple formats." Precision helps NeamCode find the right section and produce a clean edit.

Summarizing Large Files #

text
You: Give me an overview of what's in this 1200-line middleware.js

NeamCode:
  [file_read] src/middleware.js

  This file contains 14 middleware functions:

  1. authMiddleware (lines 1-45) — JWT token validation
  2. rateLimiter (lines 47-92) — Per-IP rate limiting
  3. requestLogger (lines 94-128) — Structured logging
  ...

4.6 Binary Files and Images #

NeamCode can work with more than just text files.

Reading Images #

The image_read skill lets NeamCode analyze images:

text
You: Look at the screenshot in bug-report.png and tell me what's wrong

NeamCode:
  [image_read] bug-report.png

  I can see the issue in the screenshot. The navigation bar is
  overlapping the main content area. The hamburger menu icon is
  also cut off on the right side. This looks like a CSS overflow
  issue — likely the nav container is missing `overflow: hidden`
  or the flex container needs `flex-shrink: 0` on the nav element.

Working with Binary Files #

NeamCode can detect binary files and handle them appropriately:

text
You: What's in data/export.csv?

NeamCode:
  [file_read] data/export.csv

  This CSV has 3 columns and 1,247 rows:
  - id (integer)
  - customer_name (string)
  - purchase_amount (decimal)

  First 5 rows:
  1, Alice Johnson, 49.99
  2, Bob Smith, 124.50
  ...
📝 Note

📝 Note: NeamCode can read text-based data formats (CSV, JSON, XML, YAML) just like code. For truly binary files (compiled binaries, images, archives), it will let you know it can't read the raw contents but can still help you work with them using shell commands.

4.7 Pro Tips #

1. Read Before Editing #

Always let NeamCode read a file before asking it to edit. This ensures the old_string matching will be accurate:

text
You: Read server.js then add request logging

# Better than:
You: Add request logging to server.js
# (NeamCode will read it anyway, but being explicit helps it plan)

2. Batch Related Changes #

Instead of making one edit at a time, describe the full change:

text
# Good:
You: Add a User model with validation, a users route with
     CRUD endpoints, and register the route in app.js

# Less efficient:
You: Create a User model
You: Now add a users route
You: Now register it in app.js

3. Use file_write for New Files, file_edit for Changes #

NeamCode picks the right tool automatically, but understanding the distinction helps you set expectations:

  • file_write: Creates a new file or completely overwrites an existing one
  • file_edit: Surgically modifies a specific part of an existing file

4. Trust the Approval System #

Every write and edit requires your approval. Use this as a code review step — actually read the changes before accepting. It's your last line of defense and a great way to learn.

5. When in Doubt, Ask for a Plan First #

text
You: I want to refactor the auth system. Before making changes,
     read all the relevant files and give me a plan.

This lets you review the approach before any files are touched.
