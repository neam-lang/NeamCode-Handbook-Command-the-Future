19.1 Learning a New Programming Language #

Starting a new programming language can feel overwhelming. NeamCode turns that anxiety into a guided conversation. Instead of searching Stack Overflow for hours, you can ask questions right from your terminal and get explanations tailored to what you already know.

Setting Up Your Learning Environment #

text
$ neamcode

> /mode explore

Mode switched to: Explore
Ready to explore and learn. Ask me anything.

> I'm a Python developer learning Rust. Start me with the basics
  — what's different about ownership?

NeamCode responds:

text
Great question! Coming from Python, ownership is the biggest mental
shift in Rust. Let me explain with a comparison.

In Python, you can do this freely:

    x = [1, 2, 3]
    y = x          # Both x and y point to the same list
    y.append(4)
    print(x)       # [1, 2, 3, 4] — x changed too!

In Rust, assignment *moves* ownership:

    let x = vec![1, 2, 3];
    let y = x;     // x is MOVED to y
    // println!("{:?}", x);  // ERROR! x no longer owns the data

This prevents data races and memory bugs at compile time.

Three rules to remember:
1. Each value has exactly one owner
2. When the owner goes out of scope, the value is dropped
3. You can borrow (&x) instead of moving

Want me to create a small Rust project so you can practice ownership
hands-on?

Getting Explanations for Compiler Errors #

text
> cargo run gives me this error:
  "error[E0382]: borrow of moved value: `data`"
  on line 23. Explain what this means and how to fix it.

This error means you used `data` after it was moved to another variable
or function.

Looking at line 23 in your file:

    let data = String::from("hello");
    let other = data;           // data is moved here
    println!("{}", data);       // ERROR: data was moved on line above

Fix options:
  1. Clone it:    let other = data.clone();
  2. Borrow it:   let other = &data;
  3. Use it after: swap the println before the move

For exercise 1, option 2 (borrowing) is the intended solution.

# FastCache — Project Overview

**Purpose:** An in-memory caching library for Go with LRU eviction
and TTL support.

**Structure:**
    fastcache/
    ├── cache.go          # Core Cache struct and public API
    ├── lru.go            # LRU eviction policy
    ├── ttl.go            # TTL (time-to-live) expiration
    ├── shard.go          # Sharding for concurrent access
    ├── hash.go           # FNV-based hash function
    ├── stats.go          # Hit/miss/eviction counters
    ├── cache_test.go     # Unit tests (87% coverage)
    ├── benchmark_test.go # Performance benchmarks
    └── examples/
        ├── basic/        # Simple get/set example
        └── http/         # HTTP cache middleware

**Entry Points:**
  1. cache.New(opts) — creates a new cache instance
  2. cache.Get(key) — retrieves a value
  3. cache.Set(key, value, ttl) — stores a value
  4. cache.Delete(key) — removes a value

**Architecture:**
    ┌─────────────────────────────┐
    │        Public API           │
    │   Get / Set / Delete        │
    ├─────────────────────────────┤
    │      Shard Router           │
    │   hash(key) % numShards     │
    ├──────┬──────┬──────┬────────┤
    │Shard │Shard │Shard │ Shard  │
    │  0   │  1   │  2   │  ...   │
    │ LRU  │ LRU  │ LRU  │  LRU   │
    │ TTL  │ TTL  │ TTL  │  TTL   │
    └──────┴──────┴──────┴────────┘

Each shard has its own mutex, LRU list, and TTL heap.
This design allows concurrent access without a global lock.

Tracing a Function Call #

text
> Trace what happens when I call cache.Set("user:123", userData, 5*time.Minute).
  Walk me through every function it touches.

Here's the call chain for Set():

  1. cache.go:Set(key, value, ttl)
     → Computes shard index: hash(key) % numShards

  2. shard.go:shard.set(key, value, ttl)
     → Acquires shard mutex lock
     → Checks if key already exists

  3. If key exists:
     → lru.go:lru.moveToFront(entry)
     → Updates entry.value and entry.expireAt

  4. If key is new:
     → Checks if shard is at capacity
     → If full: lru.go:lru.evictLast() removes oldest entry
     → Creates new entry, adds to front of LRU list
     → ttl.go:ttl.schedule(entry) adds to expiration heap

  5. shard.go: releases mutex lock

  6. stats.go:stats.recordSet() increments set counter

Total functions touched: 6 files, 8 functions

Asking "Why" Questions #

text
> Why does this project use sharding instead of a single mutex?
  Is the complexity worth it?

Good architectural question!

**Why sharding:**
A single mutex means only one goroutine can read/write at a time.
With sharding (default 256 shards), up to 256 goroutines can
operate concurrently on different keys.

**Is it worth it?**
Looking at their benchmarks:

  BenchmarkSet/single_mutex-8     2,400,000 ops/sec
  BenchmarkSet/256_shards-8      18,000,000 ops/sec  (7.5x faster)

  BenchmarkGet/single_mutex-8     5,100,000 ops/sec
  BenchmarkGet/256_shards-8      42,000,000 ops/sec  (8.2x faster)

For a caching library, this is absolutely worth the added ~100 lines
of sharding code. The performance difference is dramatic under
concurrent load.

19.3 Getting Homework Help (The Right Way) #

There is a right way and a wrong way to use AI for homework. The wrong way is copying answers. The right way is using AI as a tutor that helps you understand concepts so you can solve problems yourself.

The Wrong Way (Don't Do This) #

text
> Write the solution to this binary search tree assignment for me.

⚠️  I can help you learn, but writing your assignment solution directly
would be academic dishonesty. Instead, let me:

  1. Explain the concepts you need
  2. Walk through the algorithm step by step
  3. Help you debug YOUR code
  4. Point out where your logic has gaps

Which approach would you like?

The Right Way #

text
> I need to implement a binary search tree insert function in Java.
  I understand the concept but I'm stuck on the recursive case.
  Here's what I have so far...

  public void insert(int value) {
      root = insertRec(root, value);
  }

  private Node insertRec(Node node, int value) {
      if (node == null) {
          return new Node(value);
      }
      // I'm stuck here — how do I decide left vs right?
  }

NeamCode responds:

text
You're on the right track! The base case is correct. For the
recursive case, think about the BST property:

  - All values LESS than the current node go LEFT
  - All values GREATER than the current node go RIGHT

So you need two comparisons:

    if (value < node.value) {
        node.left = insertRec(node.left, value);
    } else if (value > node.value) {
        node.right = insertRec(node.right, value);
    }
    // If equal, we typically don't insert duplicates
    return node;

Notice the pattern: we recurse down the tree, and the returned
node gets assigned back to the parent's left or right pointer.
This is how the new node gets "attached" to the tree.

Try adding this to your code and test with: insert(5), insert(3),
insert(7), insert(1). Draw the tree on paper to verify.

Asking for Concept Explanations #

text
> Explain Big O notation like I'm in my first CS class.
  Use examples I can relate to.

# Big O Notation — The Simple Version

Big O tells you how your code slows down as the input gets bigger.
Think of it like this:

**O(1) — Constant: Looking up your locker**
  You know the number, you go straight there.
  Doesn't matter if the school has 100 or 10,000 lockers.
  Example: array[5] — direct index access

**O(log n) — Logarithmic: Finding a word in a dictionary**
  You open to the middle, then half again, then half again.
  A 1,000-page dictionary takes ~10 steps. 1,000,000 pages? ~20.
  Example: binary search

**O(n) — Linear: Reading every page of a book**
  Twice as many pages = twice as long.
  Example: finding the max value in an unsorted array

**O(n²) — Quadratic: Comparing every student to every other student**
  30 students = 900 comparisons. 300 students = 90,000.
  Example: bubble sort, nested loops

    O(1)  <  O(log n)  <  O(n)  <  O(n log n)  <  O(n²)
    fast  ←————————————————————————————————→  slow
⚠️ Warning

⚠️ Warning: Always check your school's academic integrity policy. Using AI to learn concepts is generally fine. Submitting AI-generated code as your own work is generally not. When in doubt, ask your professor.

> My merge sort implementation returns the wrong output.
  Input: [38, 27, 43, 3, 9, 82, 10]
  Expected: [3, 9, 10, 27, 38, 43, 82]
  Actual: [3, 9, 10, 27, 27, 38, 43]

  The number 82 disappears and 27 appears twice. Here's my code.

NeamCode responds:

text
Classic merge sort bug! The symptom (element disappearing, another
duplicating) points to an off-by-one error in your merge step.

Let me look at your merge function...

Found it! In merge_sort.py, line 34:

    while j < len(right):
        result.append(left[i])    # ← BUG: should be right[j]
        j += 1

You're appending from `left[i]` when you should be appending
from `right[j]`. This is in the cleanup loop that handles
remaining elements in the right array.

Change line 34 to:
    result.append(right[j])

This is a very common typo. The fix is one character: left → right.

Want me to walk through why this causes the specific symptom
you're seeing?

Understanding Why the Bug Causes That Symptom #

text
> Yes, walk me through it. I want to understand, not just fix it.

Let's trace the execution with your input [38, 27, 43, 3, 9, 82, 10]:

Split phase (correct):
  [38, 27, 43, 3] and [9, 82, 10]
  → [38, 27] [43, 3] [9, 82] [10]
  → [38] [27] [43] [3] [9] [82] [10]

Merge phase — the bug hits when merging [43, 82] with [10]:

  Merge [43, 82] and [10]:
    Compare 43 vs 10 → append 10, j=1
    Right array exhausted. Enter left cleanup loop:
      append 43, i=1
      append 82, i=2  ← correct so far

  But when merging [3, 10] with [43, 82]:
    Compare 3 vs 43 → append 3, i=1
    Compare 10 vs 43 → append 10, i=2
    Left array exhausted. Enter RIGHT cleanup loop:
      Line 34: append left[i] → append left[2]
                                          ↑
      left only has indices 0,1. This is index 2!
      In Python, this would be an IndexError...

  Wait — actually, looking more carefully at your code, `i`
  hasn't been incremented past the array bounds because of
  how your loop variable works. Let me re-trace...

  The issue is that left[i] still points to the last compared
  element (10), so it gets appended again instead of 82.
  That's why 27 (or whichever element) appears twice — the
  wrong array is being read during cleanup.

  Fix: change left[i] to right[j] on line 34.

19.5 Preparing for Coding Interviews #

NeamCode can simulate a coding interview environment, give you problems, evaluate your solutions, and coach you on how to communicate your thought process.

Starting an Interview Practice Session #

text
> I have a Google interview in 2 weeks. Give me a medium-difficulty
  array problem and act as my interviewer. Don't give hints unless

**Interview Question: Product of Array Except Self**

Given an integer array `nums`, return an array `answer` where
`answer[i]` is equal to the product of all elements of `nums`
except `nums[i]`.

You must solve it in O(n) time WITHOUT using division.

Example:
  Input:  [1, 2, 3, 4]
  Output: [24, 12, 8, 6]

Take your time. Talk through your approach before coding.

Working Through the Problem #

text
> Okay, my first thought is brute force — for each element, multiply
  all the others. That's O(n²). Let me think about O(n)...

  If I could use division, I'd multiply everything and divide by
  each element. But I can't.

  What if I precompute prefix products and suffix products?
  prefix[i] = product of everything before index i
  suffix[i] = product of everything after index i
  Then answer[i] = prefix[i] * suffix[i]

Good approach! A few follow-up observations:

✅ You identified brute force first and its complexity — that shows
   systematic thinking. Interviewers like this.

✅ You considered the division approach and correctly noted the
   constraint. This shows you read the problem carefully.

✅ The prefix/suffix approach is correct and achieves O(n) time.

One thing to consider: what's the space complexity of your approach?
Can you optimize it?

Go ahead and code it up. I'll evaluate when you're done.

Getting Feedback #

text
> Here's my solution:

  def productExceptSelf(nums):
      n = len(nums)
      answer = [1] * n
      prefix = 1
      for i in range(n):
          answer[i] = prefix
          prefix *= nums[i]
      suffix = 1
      for i in range(n-1, -1, -1):
          answer[i] *= suffix
          suffix *= nums[i]
      return answer

**Interview Evaluation:**

Code Correctness: ✅ Pass
  - Handles the example correctly
  - Handles edge cases (zeros, negative numbers)

Time Complexity: ✅ O(n) — two passes
Space Complexity: ✅ O(1) extra space (answer array doesn't count)

Communication: ⭐⭐⭐⭐ (4/5)
  - You walked through your thought process clearly
  - You started with brute force and optimized
  - Suggestion: explicitly state the space optimization
    ("I'm using the output array to store prefix products
     in the first pass, then multiplying suffix in-place
     in the second pass")

Code Quality: ⭐⭐⭐⭐⭐ (5/5)
  - Clean variable names
  - Concise implementation
  - Used the O(1) space optimization

Overall: Strong hire signal. Ready for this type of question.
Want another problem? I can increase difficulty.

19.7 Reproducing Research Paper Code #

You read a paper for your ML class and want to implement the algorithm. NeamCode can help bridge the gap between mathematical notation and working code.

Understanding Paper Algorithms #

text
> I'm reading "Attention Is All You Need" and I need to implement
  the scaled dot-product attention from Section 3.2.1. The formula
  is Attention(Q,K,V) = softmax(QK^T / sqrt(d_k))V

  Help me understand this and implement it in PyTorch.

Let me break down the formula into plain English:

  Attention(Q, K, V) = softmax(Q K^T / sqrt(d_k)) V

  Step by step:
  1. Q K^T       → "How much does each query match each key?"
                    Matrix multiply, gives similarity scores
  2. / sqrt(d_k) → "Scale down so softmax isn't too spiky"
                    d_k is the dimension of keys
  3. softmax()   → "Convert scores to probabilities (sum to 1)"
  4. × V         → "Weight the values by those probabilities"

  Diagram:
    Q ──┐
        ├── MatMul ── Scale ── Softmax ──┐
    K ──┘                                ├── MatMul ── Output
    V ───────────────────────────────────┘

Here's the PyTorch implementation:

  import torch
  import torch.nn.functional as F
  import math

  def scaled_dot_product_attention(Q, K, V, mask=None):
      """
      Q: (batch, heads, seq_len, d_k)
      K: (batch, heads, seq_len, d_k)
      V: (batch, heads, seq_len, d_v)
      """
      d_k = Q.size(-1)

      # Step 1 & 2: QK^T / sqrt(d_k)
      scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(d_k)

      # Optional: apply mask (for decoder self-attention)
      if mask is not None:
          scores = scores.masked_fill(mask == 0, -1e9)

      # Step 3: softmax
      attention_weights = F.softmax(scores, dim=-1)

      # Step 4: multiply by V
      output = torch.matmul(attention_weights, V)

      return output, attention_weights

Each line maps directly to a step in the formula.
Want me to explain the mask parameter or move on to
multi-head attention?

> Let me test this with small tensors to verify it's correct.

Here's a simple test case you can verify by hand:

  Q = torch.tensor([[[[1.0, 0.0],
                       [0.0, 1.0],
                       [1.0, 1.0]]]])

  K = torch.tensor([[[[1.0, 0.0],
                       [0.0, 1.0],
                       [1.0, 1.0]]]])

          = [0.585, 0.414]

  They should match within floating-point tolerance.
