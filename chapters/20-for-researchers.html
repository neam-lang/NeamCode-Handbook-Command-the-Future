20.1 Building Data Analysis Pipelines #

Research often starts with messy data that needs cleaning, transforming, and analyzing. NeamCode can help you build robust pipelines without leaving your terminal.

Starting a Data Analysis Project #

text
$ cd ~/research/climate-study
$ neamcode

> /mode analyze

Mode switched to: Analyze
Ready to analyze code structure and patterns.

> I have a CSV file with 10 years of daily temperature readings
  from 50 weather stations. Columns: station_id, date, temp_min,
  temp_max, temp_avg, precipitation, humidity. Some rows have
  missing values. Help me build a cleaning and analysis pipeline.

NeamCode responds:

text
I'll help you build a robust pipeline. Let me first look at
the data to understand what we're working with.

Looking at weather_data.csv...

File summary:
  - 182,500 rows (50 stations × 365 days × 10 years)
  - 7 columns as described
  - Missing values: temp_avg (2.3%), precipitation (5.1%),
    humidity (1.8%)
  - Date range: 2014-01-01 to 2023-12-31

I'll create a pipeline with these stages:

  Raw CSV
    │
    ▼
  1. Load & Validate (check types, ranges)
    │
    ▼
  2. Clean (handle missing, remove outliers)
    │
    ▼
  3. Transform (add derived columns, resample)
    │
    ▼
  4. Analyze (statistics, trends, correlations)
    │
    ▼
  5. Export (cleaned CSV + summary stats)

Created: pipeline.py

Key decisions:
  - Using pandas for data manipulation
  - Missing temp_avg: interpolated from temp_min/temp_max
  - Missing precipitation: forward-fill (weather patterns)
  - Outlier detection: values beyond 3 standard deviations
    flagged, not removed (you decide)

Iterating on Analysis #

text
> The outlier detection is too aggressive. Station 23 is in Death Valley
  — its high temps are legitimate, not outliers. How should I handle
  geographic variation?

Good catch. Per-station statistics would be better than global ones.

Updated the outlier detection in pipeline.py:

  Before (global):
    mean = df['temp_max'].mean()
    std = df['temp_max'].std()
    outliers = df[abs(df['temp_max'] - mean) > 3 * std]

  After (per-station):
    station_stats = df.groupby('station_id')['temp_max'].agg(['mean', 'std'])
    df = df.merge(station_stats, on='station_id')
    df['is_outlier'] = abs(df['temp_max'] - df['mean']) > 3 * df['std']

This way, Death Valley's 54°C readings are normal for *that station*,
while a 54°C reading in Alaska would still be flagged.

Per-station outlier counts:
  Station 23 (Death Valley): 2 outliers (down from 847)
  Station 7 (Fairbanks): 5 outliers (unchanged)
  ...

Summarizing "A Survey of Efficient Attention Mechanisms"...

**Taxonomy of Efficient Attention (from Section 3):**

  Efficient Attention
  ├── Sparse Attention
  │   ├── Fixed patterns (local window, strided)
  │   ├── Learned patterns (routing, clustering)
  │   └── Combined (BigBird, Longformer)
  ├── Linear Attention
  │   ├── Kernel methods (Performers, RFA)
  │   ├── State-space models (Mamba, S4)
  │   └── Hybrid (Hyena, RWKV)
  ├── IO-Aware Attention
  │   ├── Tiling-based (FlashAttention family)
  │   └── Hardware-specific (ring attention)
  └── Approximate Attention
      ├── Low-rank (Linformer, Nyströmformer)
      ├── Hashing (Reformer, SMYRF)
      └── Sampling-based (Random Feature Attention)

**Key finding:** No single approach dominates. Choice depends on:
  - Sequence length (linear better for >8K tokens)
  - Hardware (IO-aware critical for GPU, less for TPU)
  - Task type (sparse good for local tasks, linear for global)

**Their recommendation:** FlashAttention for <8K, Mamba for >32K,

20.3 Automating Experiments #

Running the same experiment with different parameters, collecting results, and comparing them is tedious. NeamCode can help you automate the entire loop.

Setting Up an Experiment Framework #

text
> I need to run my neural network training with different
  hyperparameters. I want to test:
  - Learning rates: [0.001, 0.0005, 0.0001]
  - Batch sizes: [32, 64, 128]
  - Dropout: [0.1, 0.3, 0.5]
  That's 27 combinations. Help me set up a sweep.

I'll create an experiment runner that handles the full sweep.

Created: experiments/sweep.py

Features:
  - Generates all 27 parameter combinations
  - Runs each as a subprocess with timeout
  - Logs results to experiments/results.jsonl
  - Saves best model checkpoint automatically
  - Resumes from where it left off if interrupted
  - Estimated time: ~27 × [your training time per run]

Configuration (experiments/sweep_config.yaml):

  experiment_name: "hyperparam_sweep_v1"
  base_command: "python train.py"
  parameters:
    learning_rate: [0.001, 0.0005, 0.0001]
    batch_size: [32, 64, 128]
    dropout: [0.1, 0.3, 0.5]
  metrics:
    primary: "val_accuracy"
    secondary: ["val_loss", "train_loss"]
  resources:
    gpu: 1
    timeout_minutes: 30

Run with: python experiments/sweep.py

To monitor progress:
  tail -f experiments/results.jsonl | python experiments/live_plot.py

Analyzing Experiment Results #

text
> The sweep finished. Here are the results in results.jsonl.
  Analyze them and tell me the best configuration and any
  interesting patterns.

Analyzing 27 experiment runs...

**Best Configuration:**
  Learning rate: 0.0005
  Batch size: 64
  Dropout: 0.3
  Val accuracy: 94.2%  |  Val loss: 0.187

**Top 5 Results:**
  ┌────┬────────┬───────┬─────────┬──────────┬──────────┐
  │ #  │   LR   │ Batch │ Dropout │ Val Acc  │ Val Loss │
  ├────┼────────┼───────┼─────────┼──────────┼──────────┤
  │ 1  │ 0.0005 │   64  │   0.3   │  94.2%   │  0.187   │
  │ 2  │ 0.0005 │   64  │   0.1   │  93.8%   │  0.195   │
  │ 3  │ 0.001  │   64  │   0.3   │  93.5%   │  0.201   │
  │ 4  │ 0.0005 │   32  │   0.3   │  93.1%   │  0.211   │
  │ 5  │ 0.0001 │   64  │   0.3   │  92.9%   │  0.218   │
  └────┴────────┴───────┴─────────┴──────────┴──────────┘

**Patterns observed:**

  1. Batch size 64 dominates the top results (4 of top 5).
     Batch 128 underperformed — likely too large for dataset size.

  2. Learning rate sensitivity:
     0.001: avg 91.3% (some runs unstable)
     0.0005: avg 93.2% (consistent, best)
     0.0001: avg 90.8% (underfitting, needs more epochs)

  3. Dropout sweet spot at 0.3:
     0.1: slight overfitting (train-val gap: 4.2%)
     0.3: balanced (train-val gap: 1.8%)
     0.5: underfitting (train acc also drops)

  4. Three runs with LR=0.001 + batch=128 diverged (NaN loss).
     This combination is unstable.

**Recommendation:** Use LR=0.0005, batch=64, dropout=0.3.
Consider a finer sweep around these values: LR [0.0003-0.0007],
dropout [0.25-0.35].

20.5 Reproducible Research Environments #

The "it works on my machine" problem kills reproducibility. NeamCode helps you create environments that anyone can recreate.

Creating a Reproducible Setup #

text
> Help me make this project fully reproducible. Someone should be
  able to clone my repo and reproduce all results from my paper.

I'll set up a complete reproducibility stack:

Created/updated files:

  1. environment.yml (Conda environment):
     - Python 3.11.7 (exact version)
     - All packages pinned to exact versions
     - CUDA 11.8 for GPU support

  2. Dockerfile:
     - Based on nvidia/cuda:11.8.0-runtime-ubuntu22.04
     - Installs exact conda environment
     - Sets random seeds in entrypoint

  3. Makefile:
     - make setup     → create environment
     - make data      → download/preprocess data
     - make train     → run training
     - make evaluate  → run evaluation
     - make figures   → generate all paper figures
     - make all       → full pipeline

  4. reproduce.sh:
     - One-command reproduction script
     - Downloads data, trains, evaluates, generates figures
     - Logs everything to reproduce.log
     - Sets PYTHONHASHSEED, numpy seed, torch seed

  5. data/README.md:
     - Data sources with download URLs
     - SHA256 checksums for verification
     - Preprocessing steps documented

  Reproducibility checklist:
    ✅ Exact package versions (environment.yml)
    ✅ Random seeds set everywhere (reproduce.sh)
    ✅ Data checksums (data/README.md)
    ✅ Hardware requirements documented (README.md)
    ✅ One-command reproduction (make all)
    ✅ Docker option for exact environment (Dockerfile)
    ⬜ Pre-trained model checkpoint (upload to Zenodo?)

20.6 Using Neam for Agent Research #

If your research involves AI agents, the Neam programming language itself is a powerful tool for rapid prototyping of multi-agent systems.

Prototyping a Multi-Agent Experiment #

text
> I'm researching multi-agent debate for improving LLM reasoning.
  I need two agents that debate a question and a judge agent that
  evaluates their arguments. Can NeamCode help me prototype this
  in the Neam language?

Absolutely. Neam has first-class agent support. Here's a prototype:

Created: debate_experiment.neam

  budget DebateBudget {
      cost: 1.00,
      tokens: 50000,
      time: 120000
  }

  agent Debater_A {
      provider: "openai",
      model: "gpt-4o-mini",
      system: "You argue FOR the given position. Be concise
               and logical. Cite evidence. Max 150 words.",
      temperature: 0.7,
      budget: DebateBudget
  }

  agent Debater_B {
      provider: "openai",
      model: "gpt-4o-mini",
      system: "You argue AGAINST the given position. Be concise
               and logical. Cite evidence. Max 150 words.",
      temperature: 0.7,
      budget: DebateBudget
  }

  agent Judge {
      provider: "openai",
      model: "gpt-4o",
      system: "You are an impartial judge. Read both arguments.
               Score each 1-10 on: logic, evidence, persuasion.
               Declare a winner with reasoning.",
      temperature: 0.3,
      budget: DebateBudget
  }

  fun debate(question: str, rounds: int) {
      var history = []
      for round in 1..rounds {
          let arg_a = ask Debater_A: question + "\n\nPrevious:\n" + history
          let arg_b = ask Debater_B: question + "\n\nPrevious:\n" + history
          history.append("Round {round}:\nFOR: {arg_a}\nAGAINST: {arg_b}")
      }
      let verdict = ask Judge: "Question: {question}\n\n{history.join('\n')}"
      return verdict
  }

  // Run experiment
  let result = debate("Should AI systems be open-sourced?", rounds: 3)
  print(result)

This gives you a working debate framework. You can:
  - Vary the number of rounds
  - Swap models to compare providers
  - Log all arguments for analysis
  - Run batch experiments with different questions

Running Batch Experiments with Cost Tracking #

text
> I need to run this debate on 50 different questions and compare
  results with 2 rounds vs 4 rounds. Track costs carefully.

Updated: debate_experiment.neam with batch runner.

  budget ExperimentBudget {
      cost: 25.00,       // Total budget for all 100 runs
      tokens: 2500000,
      time: 600000
  }

  fun run_batch(questions: list, round_counts: list) {
      var results = []
      for q in questions {
          for rounds in round_counts {
              let result = debate(q, rounds)
              results.append({
                  question: q,
                  rounds: rounds,
                  verdict: result,
                  cost: get_cost()  // Neam built-in cost tracking
              })
          }
      }
      save_json("results/debate_results.json", results)
  }

  let questions = load_json("questions.json")
  run_batch(questions, [2, 4])

Run with: neam debate_experiment.neam

Cost tracking output:
  Total: 100 debates
  Budget used: $18.42 / $25.00
  Average per debate: $0.18
  2-round avg cost: $0.12  |  4-round avg cost: $0.24
