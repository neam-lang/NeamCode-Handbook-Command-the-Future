14.1 Why Run Locally? #

Before diving into setup, let's be clear about why you'd want this.

Cost: $0.00. Every token generated by a local model is free. No API keys, no billing accounts, no surprise invoices.

Privacy: Total. Your code, prompts, and conversations never leave your machine. No data is sent to any server. This matters for proprietary code, client projects, and regulated industries.

Speed: No network round-trip. On capable hardware, local inference can match or beat cloud latency, especially when factoring in network overhead.

Availability: Always on. No outages, no rate limits, no API deprecations. Your AI assistant works on airplanes, in basements, and during cloud provider incidents.

The trade-off? Local models are smaller and less capable than frontier cloud models. A 7B parameter model running on your laptop won't match Claude Opus or GPT-4o on complex reasoning tasks. But for 80% of everyday coding tasks — completions, refactoring, test generation, explanations — local models are surprisingly capable.

14.2 Installing Ollama #

Ollama is the local AI runtime that NeamCode uses. Installation is straightforward.

macOS #

bash
# Install via Homebrew
brew install ollama

# Start the service
ollama serve

Or download the desktop app from ollama.com which includes a menu bar icon and auto-start.

Linux #

bash
# One-line install
curl -fsSL https://ollama.com/install.sh | sh

# Start the service
sudo systemctl start ollama
sudo systemctl enable ollama   # auto-start on boot

Verify Installation #

bash
# Check Ollama is running
ollama --version
# ollama version 0.3.x

# Check the API endpoint
curl http://localhost:11434/api/version
# {"version":"0.3.x"}

Pull Your First Model #

bash
# A good starting model for most machines
ollama pull qwen2.5-coder:7b

# Test it works
ollama run qwen2.5-coder:7b "Write a Python hello world"
💡 Tip

💡 Tip: The first time you pull a model, it downloads several gigabytes. Do this on a good connection before you need offline capability.

14.3 Choosing the Right Model for Your Machine #

This is the most important decision in your local setup. Choosing a model too large for your hardware leads to painfully slow responses. Choosing one too small sacrifices quality unnecessarily.

14.4 Configuring NeamCode for Local-First #

To make NeamCode default to local models, configure it as your primary provider:

yaml
# neamcode.config.yml — Local-first configuration

providers:
  ollama:
    endpoint: "http://localhost:11434"
    enabled: true
    default_model: "qwen2.5-coder:7b"

routing:
  strategy: "cost"          # prefers free/cheap options
  fallback_chain:
    - ollama                # local first

That's the minimal "pure local" setup. No API keys needed. No accounts. No billing.

Verify it's working:

text
neamcode $ /status

Provider Status:
  ● Ollama       — connected (qwen2.5-coder:7b, local)

Routing:
  Strategy:        cost
  Active provider: Ollama (local)
  Session cost:    $0.00

neamcode $ Write a function that reverses a linked list in Python

Using Ollama (qwen2.5-coder:7b) — local, $0.00

Here's an implementation of linked list reversal:

def reverse_linked_list(head):
    prev = None
    current = head
    while current:
        next_node = current.next
        current.next = prev
        prev = current
        current = next_node
    return prev

Configure the Hybrid Setup #

yaml
# neamcode.config.yml — Hybrid 80/15/5 configuration

providers:
  ollama:
    endpoint: "http://localhost:11434"
    enabled: true
    default_model: "qwen2.5-coder:7b"

  anthropic:
    api_key: "${ANTHROPIC_API_KEY}"
    enabled: true
    default_model: "claude-haiku"       # cheap cloud tier

  openai:
    api_key: "${OPENAI_API_KEY}"
    enabled: true
    default_model: "gpt-4o"            # premium tier

routing:
  strategy: "balanced"
  fallback_chain:
    - ollama          # free, always first
    - anthropic       # cheap cloud fallback
    - openai          # premium when needed

tokens:
  max_output: 8192
  warning_threshold: 0.8

How It Works in Practice #

NeamCode's routing engine assesses each task and routes accordingly:

text
neamcode $ What's the syntax for a Python list comprehension?

Using Ollama (qwen2.5-coder:7b) — local, $0.00
  [simple factual question → local model handles it fine]

neamcode $ Explain why this async code is deadlocking

Using Anthropic (claude-haiku) — $0.002
  [medium complexity → small cloud model for better reasoning]

neamcode $ /provider anthropic claude-opus
neamcode $ Review this authentication system for security vulnerabilities

Using Anthropic (claude-opus) — $0.18
  [critical task → manually selected premium model]

neamcode $ /routing balanced
  [back to auto-routing]
💡 Tip

💡 Tip: You don't have to manually route every request. The balanced strategy handles most routing decisions well. Only override manually for critical tasks where you specifically want premium quality.

14.7 Quality Expectations #

Let's be honest about what local models can and cannot do well.

Tasks Local Models Handle Well (80% of your work) #

  • Code completions and boilerplate generation
  • Simple refactoring (rename, extract function, change patterns)
  • Writing unit tests for straightforward functions
  • Explaining error messages and stack traces
  • Generating documentation and comments
  • Language syntax questions
  • Simple bug fixes
  • File operations and git commands
text
neamcode $ Using Ollama (qwen2.5-coder:7b)

You: Add type hints to this function

def process_data(items, threshold, callback):
    results = []
    for item in items:
        if item.value > threshold:
            results.append(callback(item))
    return results

NeamCode: Here's the function with type hints added:

from typing import List, Callable

def process_data(
    items: List[DataItem],
    threshold: float,
    callback: Callable[[DataItem], Result]
) -> List[Result]:
    results: List[Result] = []
    for item in items:
        if item.value > threshold:
            results.append(callback(item))
    return results

  ✓ Quality: Correct and well-formatted

Tasks Where Cloud Models Shine (20% of your work) #

  • Complex architectural decisions
  • Subtle bug analysis (race conditions, memory leaks)
  • Security vulnerability assessment
  • Multi-file refactoring with deep understanding
  • Nuanced code review with business context
  • Novel algorithm design
  • Understanding complex legacy code
text
neamcode $ Using Ollama (qwen2.5-coder:7b)

You: Is there a race condition in this concurrent map implementation?

NeamCode: The implementation looks correct. The mutex protects
  the map access.

neamcode $ /provider anthropic claude-opus

You: Is there a race condition in this concurrent map implementation?

NeamCode: Yes, there's a subtle race condition. While the mutex
  protects individual map operations, the read-check-write pattern
  in the GetOrCreate method isn't atomic:

  1. Thread A reads: key not found
  2. Thread B reads: key not found
  3. Thread A writes: creates value
  4. Thread B writes: overwrites with duplicate

  You need to hold the lock across the entire GetOrCreate operation,
  or use sync.Map with LoadOrStore for this pattern.

  ✓ Cloud model caught the subtle issue the local model missed

14.8 Pro Tips #

💡 Tip

💡 Tip: Pull the coding-specific model. qwen2.5-coder:7b consistently outperforms general-purpose models of the same size on coding tasks. It's the best bang-for-your-buck local model for NeamCode.

💡 Tip

💡 Tip: Keep models warm. Ollama unloads models from memory after inactivity. The first request after idle requires loading the model (5-30 seconds). If you're actively coding, this rarely matters since the model stays loaded.

💡 Tip

💡 Tip: Use the cost routing strategy as your default. With the hybrid setup, cost routing sends 80%+ of requests to your free local model. You'll barely notice the quality difference on routine tasks.

yaml
routing:
  strategy: "cost"       # local first, cloud when needed
💡 Tip

💡 Tip: Monitor your actual usage. Run /cost at the end of each week to see your real spending pattern. Most developers are surprised how little they actually need cloud models.

text
neamcode $ /cost

Session Cost Report:
  Total:           $2.15
  Local requests:  847  ($0.00)
  Cloud requests:  153  ($2.15)
  Local ratio:     85%

  At this rate: ~$9.46/month
💡 Tip

💡 Tip: SSD vs. HDD matters. Model loading speed depends on storage. If you're on an HDD, the initial model load will be noticeably slower. SSDs (especially NVMe) make local AI feel much more responsive.

💡 Tip

💡 Tip: Apple Silicon users have an advantage. M-series chips have unified memory architecture, meaning GPU and CPU share the same RAM pool. A 16GB M2 MacBook runs 7B models faster than many dedicated GPU setups. NeamCode and Ollama take full advantage of this.

💡 Tip

💡 Tip: Multiple models for different tasks. Pull a fast model (phi3) for quick questions and a capable model (qwen2.5-coder:7b) for real coding tasks. Switch with /provider ollama phi3 when you just need a fast answer.

⚠️ Warning

⚠️ Warning: Be realistic about 8GB machines. They can run local models, but you'll be limited to smaller models and may experience memory pressure if running an IDE, browser, and NeamCode simultaneously. Consider closing unnecessary applications during intensive local AI sessions.
