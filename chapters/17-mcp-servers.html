Chapter 17: MCP Servers — Extending NeamCode #

"Any sufficiently advanced tool is indistinguishable from an ecosystem."

What You'll Learn #

  • How to manage your MCP connections
  • How to expose NeamCode itself as an MCP server for other tools
  • Pro tips for working with MCP effectively

17.1 What Is MCP? #

The Model Context Protocol (MCP) is an open standard that lets AI assistants connect to external tools and services through a unified interface. Think of it as a USB port for AI — any tool that speaks MCP can plug into any AI assistant that supports it.

text
┌──────────────┐       MCP        ┌──────────────────┐
│   NeamCode   │◄────────────────►│  Notion Server   │
│              │                  └──────────────────┘
│              │       MCP        ┌──────────────────┐
│              │◄────────────────►│  GitHub Server   │
│              │                  └──────────────────┘
│              │       MCP        ┌──────────────────┐
│              │◄────────────────►│  Slack Server    │
│              │                  └──────────────────┘
│              │       MCP        ┌──────────────────┐
│              │◄────────────────►│  Database Server │
└──────────────┘                  └──────────────────┘

Without MCP, you'd need custom integrations for every tool. With MCP, any server that implements the protocol automatically works with NeamCode.

What MCP Servers Provide #

An MCP server can expose:

  • Tools — Functions NeamCode can call (e.g., "create a Jira ticket", "query a database", "send a Slack message")
  • Resources — Data NeamCode can read (e.g., database schemas, document contents)
  • Prompts — Pre-built prompt templates for common tasks

How NeamCode Connects #

NeamCode acts as an MCP client. It connects to one or more MCP servers, discovers what tools they offer, and makes those tools available during your session. You configure which servers to connect to, and NeamCode handles the rest.

17.2 Connecting an MCP Server #

MCP servers are configured in your NeamCode settings. There are two common transport types: stdio (local process) and SSE/HTTP (remote).

Local Server (stdio) #

A local MCP server runs as a subprocess. NeamCode starts it, communicates over stdin/stdout, and stops it when you disconnect.

Add it to your configuration file (.neamcode/settings.json or global config):

json
{
  "mcp_servers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/projects"],
      "transport": "stdio"
    }
  }
}

Remote Server (SSE) #

A remote MCP server runs on a network endpoint:

json
{
  "mcp_servers": {
    "company-tools": {
      "url": "https://mcp.internal.company.com/sse",
      "transport": "sse",
      "headers": {
        "Authorization": "Bearer ${MCP_COMPANY_TOKEN}"
      }
    }
  }
}

Verifying the Connection #

After configuring, verify that NeamCode can reach the server:

text
You: /mcp list
text
MCP Server Connections
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  filesystem       stdio       ✅ Connected   8

💡 Tip

💡 Tip: You can use environment variables in your MCP configuration (like ${MCP_COMPANY_TOKEN} above). NeamCode resolves them at connection time, so you never hardcode secrets.

17.3 Using MCP Tools #

Once a server is connected, its tools are available to NeamCode automatically. You can use them in two ways: implicitly (just describe what you want) or explicitly (browse and call tools directly).

Implicit Use (Recommended) #

Just tell NeamCode what you need. It knows which MCP tools are available and will use them when appropriate:

text
You: Create a new page in Notion called "Sprint 24 Planning"
     with a table for tracking tasks
text
Using tool: notion.create_page
  title: "Sprint 24 Planning"
  content: [table block with columns: Task, Owner, Status, Due Date]

✅ Page created: https://notion.so/Sprint-24-Planning-abc123

You didn't need to know the tool name or its parameters — NeamCode figured it out.

Explicit Tool Discovery #

To see what tools a server provides:

text
You: /mcp tools filesystem
text
Tools from 'filesystem' server:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  read_file        Read the contents of a file
  write_file       Write content to a file
  list_directory   List files in a directory
  create_dir       Create a new directory
  move_file        Move or rename a file
  search_files     Search for files by pattern
  get_file_info    Get metadata about a file
  delete_file      Delete a file

8 tools available

Testing a Tool #

Before relying on a tool in a workflow, you can test it:

text
You: /mcp test filesystem read_file
text
Testing: filesystem.read_file

  Sending test request with sample parameters...
  Response received in 23ms
  ✅ Tool is working correctly

  Sample output:
  {
    "content": "# README\nWelcome to the project...",
    "metadata": { "size": 1024, "modified": "2026-03-06T..." }
  }

Lazy Tool Loading #

NeamCode uses lazy tool loading for MCP connections. This means:

  • Tool definitions are fetched when you first need them, not at startup
  • Connecting many MCP servers doesn't slow down your session
  • Tools are cached after first load
text
First use:
You: Read the file /tmp/notes.txt using the filesystem server

  Loading tools from 'filesystem'... (first time, ~200ms)
  Using tool: filesystem.read_file
  ✅ Done

Second use:
You: Now list the /tmp directory

  Using tool: filesystem.list_directory  (instant — already loaded)
  ✅ Done
📝 Note

📝 Note: Lazy loading means /mcp tools <name> triggers a tool fetch if it hasn't happened yet. This is normal and only takes a moment.

17.4 Managing Connections #

Listing All Servers #

text
You: /mcp list

Shows all configured servers, their connection status, and tool counts.

Connection Troubleshooting #

If a server shows as disconnected:

text
You: /mcp list
text
MCP Server Connections
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Name             Transport   Status           Tools
  ─────────────    ─────────   ──────           ─────
  filesystem       stdio       ✅ Connected        8
  company-tools    sse         ❌ Disconnected     —

Common causes and fixes:

Problem Solution
Server binary not found Check that the command path is correct and the binary is installed
Authentication failed Verify tokens/credentials in environment variables
Network unreachable Check URL and network connectivity for SSE servers
Server crashed on start Run the server command manually to see error output

Adding a New Server Mid-Session #

You can add MCP servers to your config while NeamCode is running:

text
You: Add an MCP server for GitHub. Use the official
     @modelcontextprotocol/server-github package.

NeamCode will update your configuration and connect to the new server without restarting.

⚠️ Warning

⚠️ Warning: Be cautious when connecting to untrusted MCP servers. An MCP server's tools can read and write data on your behalf. Only connect to servers you trust or have audited.

17.5 NeamCode as an MCP Server #

NeamCode doesn't just consume MCP services — it can act as one. This means other MCP-compatible tools can call NeamCode's capabilities.

Starting NeamCode in MCP Server Mode #

bash
$ neamcode mcp-server
text
NeamCode MCP Server started
  Transport: stdio
  Tools exposed: 15
  Ready for connections.

What This Enables #

When NeamCode runs as an MCP server, other tools can:

  • Ask NeamCode to analyze code
  • Request code generation
  • Trigger NeamCode's agents (Explore, Plan, Review, Test)
  • Use NeamCode's file editing capabilities

Use Case: IDE Integration #

Your IDE can connect to NeamCode as an MCP server:

text
┌──────────────┐       MCP        ┌──────────────────┐
│   Your IDE   │────────────────►│  NeamCode         │
│   (client)   │                 │  (MCP server)     │
│              │◄────────────────│                    │
│  "Explain    │  Response:      │  Analyzes code,   │
│   this code" │  explanation    │  returns answer    │
└──────────────┘                  └──────────────────┘

Use Case: Chaining AI Assistants #

Another AI assistant can delegate coding tasks to NeamCode:

text
┌──────────────┐       MCP        ┌──────────────────┐
│  Other AI    │────────────────►│  NeamCode         │
│  Assistant   │                 │  (MCP server)     │
│              │                 │                    │
│  "Implement  │                 │  Writes code,     │
│   feature X" │◄────────────────│  runs tests,      │
│              │  Result: done   │  returns result    │
└──────────────┘                  └──────────────────┘

Configuration for MCP Server Mode #

You can customize what NeamCode exposes:

json
{
  "mcp_server": {
    "expose_tools": ["read_file", "edit_file", "run_command",
                     "agent_explore", "agent_review"],
    "allow_write": true,
    "working_directory": "/home/user/projects/myapp"
  }
}
💡 Tip

💡 Tip: When running NeamCode as an MCP server, limit expose_tools to only what the connecting client needs. Follow the principle of least privilege.

17.6 Pro Tips #

Combine Multiple MCP Servers in One Task #

NeamCode can use tools from different servers in a single conversation:

text
You: Look up the latest bug reports in Jira, find the related code
     in our repo, and post a summary to the #bugs Slack channel

NeamCode seamlessly uses tools from three servers: Jira, filesystem, and Slack — all in one flow.

Create Project-Specific MCP Configs #

Put MCP configuration in your project's .neamcode/settings.json so every team member gets the same tool access:

json
{
  "mcp_servers": {
    "our-database": {
      "command": "npx",
      "args": ["-y", "@company/db-mcp-server"],
      "transport": "stdio",
      "env": {
        "DB_HOST": "localhost",
        "DB_NAME": "myapp_dev"
      }
    }
  }
}

Check this into version control (without secrets) so the team shares a consistent setup.

Monitor Tool Usage #

Keep an eye on which MCP tools are being called, especially for tools that have side effects:

text
You: /mcp list

Review the tools column. If a server has many tools, use /mcp tools <name> to understand what you're exposing to the AI.

Build Your Own MCP Server #

If your team has internal tools or APIs, consider wrapping them in an MCP server. The protocol is straightforward:

  1. Accept tool call requests via stdio or HTTP
  2. Return structured JSON responses
  3. Provide tool descriptions so AI clients know what's available

Many teams build MCP servers for:

  • Internal databases (read-only query access)
  • Deployment systems (trigger deploys, check status)
  • Monitoring dashboards (fetch metrics, alerts)
  • Documentation systems (search internal wikis)
📝 Note

📝 Note: The MCP specification is open and well-documented. Building a simple MCP server for a REST API can be done in an afternoon.
