Chapter 12: Choosing Your AI Provider #

"The best tool is the one that fits the job. The best AI provider is the one that fits your task."
— NeamCode Philosophy

📖 Estimated read time: 18 minutes

What You'll Learn #

  • The 11 AI providers NeamCode supports and what each excels at
  • How to configure multiple providers and API keys
  • How NeamCode's routing engine automatically picks the best provider
  • The four routing strategies: quality, speed, cost, and balanced
  • How to inspect available models with /models
  • How to manually switch providers mid-session
  • Full YAML configuration reference for providers

12.1 Available Providers #

NeamCode ships with built-in support for 11 AI providers. You don't need to use all of them — most users start with one or two and expand as needed.

Which should you start with?

  • Anthropic (Claude) — Best for nuanced coding tasks, long-context analysis, and careful reasoning. The 200K context window is excellent for working with large codebases.
  • OpenAI (GPT-4o) — Strong general-purpose choice. Fast responses, wide ecosystem.
  • Gemini — Unmatched context window (1M+ tokens). Ideal if you need to feed in entire repositories at once.
  • Ollama — Free, local, private. Essential for offline work and cost-conscious setups (see Chapter 14).
  • DeepSeek — Competitive quality at lower cost. Good budget option for cloud.
  • Mistral — Strong European provider with good multilingual support.
💡 Tip

💡 Tip: You don't have to choose just one. NeamCode's routing engine can use multiple providers simultaneously, picking the best one for each request.

12.2 Setting Up Multiple Providers #

To use a provider, you need to give NeamCode its API key. Open your configuration file:

text
neamcode $ neamcode config edit

Or edit neamcode.config.yml directly. Here's how to set up three providers:

yaml
# neamcode.config.yml
providers:
  anthropic:
    api_key: "sk-ant-your-key-here"
    enabled: true
    default_model: "claude-sonnet"

  openai:
    api_key: "sk-your-openai-key-here"
    enabled: true
    default_model: "gpt-4o"

  ollama:
    endpoint: "http://localhost:11434"
    enabled: true
    default_model: "llama3.1:8b"
📝 Note

📝 Note: API keys can also be set via environment variables. NeamCode checks ANTHROPIC_API_KEY, OPENAI_API_KEY, and similar variables automatically. Environment variables override the config file.

bash
# Set via environment (recommended for security)
export ANTHROPIC_API_KEY="sk-ant-your-key-here"
export OPENAI_API_KEY="sk-your-openai-key-here"

To verify your providers are configured correctly:

text
neamcode $ /status

Provider Status:
  ● Anthropic    — connected (claude-sonnet)
  ● OpenAI       — connected (gpt-4o)
  ● Ollama       — connected (llama3.1:8b, local)
  ○ Gemini       — not configured
  ○ DeepSeek     — not configured
  ○ Mistral      — not configured
⚠️ Warning

⚠️ Warning: Never commit API keys to version control. Use environment variables or keep neamcode.config.yml in your .gitignore.

12.3 How NeamCode Chooses a Provider #

When you send a message, NeamCode doesn't just blindly forward it to one provider. It runs a routing decision based on:

text
┌──────────────────────────────────────────────┐
│              Routing Decision                 │
│                                              │
│  Your message                                │
│       │                                      │
│       ▼                                      │
│  ┌─────────────┐                             │
│  │  Routing     │◄── Strategy (quality/      │
│  │  Engine      │    speed/cost/balanced)     │
│  └──────┬──────┘                             │
│         │                                    │
│         ├── Task complexity estimate          │
│         ├── Provider availability             │
│         ├── Current cost / budget             │
│         ├── Context window needs              │
│         └── Fallback chain                    │
│         │                                    │
│         ▼                                    │
│  Selected: Anthropic (claude-sonnet)         │
└──────────────────────────────────────────────┘

The key factors:

  1. Routing strategy — Your configured preference (see 12.4)
  2. Task complexity — Simple questions may use a cheaper/faster model; complex tasks get the premium model
  3. Availability — If a provider is down, NeamCode falls back automatically
  4. Budget — If you're near your spending limit, NeamCode prefers cheaper providers
  5. Context needs — Very large contexts may route to Gemini (1M+) or Anthropic (200K)

12.4 Routing Strategies #

NeamCode supports four routing strategies. Set yours in the config:

yaml
routing:
  strategy: "balanced"  # Options: quality, speed, cost, balanced

quality (default) #

Always picks the most capable model available. Best for complex coding tasks.

text
neamcode $ /routing quality

Routing strategy set to: quality
  Priority: Claude Opus > GPT-4o > Claude Sonnet > Gemini Pro

speed #

Picks the fastest-responding model. Great for rapid iteration and quick questions.

text
neamcode $ /routing speed

Routing strategy set to: speed
  Priority: GPT-4o-mini > Claude Haiku > Gemini Flash > local Ollama

cost #

Minimizes spending. Uses local models first, then cheap cloud models.

text
neamcode $ /routing cost

Routing strategy set to: cost
  Priority: Ollama (free) > DeepSeek > GPT-4o-mini > Claude Haiku

balanced #

A mix of all three factors. The default for most users.

text
neamcode $ /routing balanced

Routing strategy set to: balanced
  Weighs quality (40%), speed (30%), cost (30%)
💡 Tip

💡 Tip: You can change your routing strategy at any time during a session with the /routing command. No restart needed.

Fallback Chains #

If your primary provider fails (API error, rate limit, outage), NeamCode automatically falls through to the next provider:

text
Primary (Anthropic) ──fail──▶ Secondary (OpenAI) ──fail──▶ Tertiary (Ollama)

Configure fallbacks explicitly:

yaml
routing:
  strategy: "balanced"
  fallback_chain:
    - anthropic
    - openai
    - ollama

When a fallback triggers, you'll see it in the status bar:

text
neamcode $ Refactor this function to use async/await

⚠ Anthropic unavailable, routing to OpenAI (gpt-4o)...

Sure, here's the refactored function using async/await:
...

12.6 Switching Providers Manually #

Sometimes you want to force a specific provider for a task. Use the /provider command:

text
neamcode $ /provider anthropic claude-opus

Switched to: Anthropic (claude-opus)
Routing: manual override (use /routing to restore auto)

This is useful when:

  • You want the best possible answer for a critical task
  • You're testing how different models handle the same prompt
  • You need a specific context window size

To return to automatic routing:

text
neamcode $ /routing balanced

Routing strategy restored: balanced
Manual override cleared.

Quick-switch examples:

text
# Use the cheapest option for a quick question
neamcode $ /provider ollama
Switched to: Ollama (llama3.1:8b, local)

# Use Gemini for a massive file analysis
neamcode $ /provider gemini
Switched to: Gemini (gemini-pro, 1M context)

# Back to auto
neamcode $ /routing quality
Routing strategy restored: quality

12.7 Provider Configuration in YAML #

Here's a comprehensive neamcode.config.yml with all provider options:

yaml
# neamcode.config.yml — Full provider configuration

providers:
  # ── Cloud Providers ──────────────────────────────
  anthropic:
    api_key: "${ANTHROPIC_API_KEY}"    # env var reference
    enabled: true
    default_model: "claude-sonnet"
    max_tokens: 8192
    temperature: 0.7

  openai:
    api_key: "${OPENAI_API_KEY}"
    enabled: true
    default_model: "gpt-4o"
    max_tokens: 4096

  gemini:
    api_key: "${GEMINI_API_KEY}"
    enabled: true
    default_model: "gemini-pro"

  deepseek:
    api_key: "${DEEPSEEK_API_KEY}"
    enabled: true
    default_model: "deepseek-v3"

  mistral:
    api_key: "${MISTRAL_API_KEY}"
    enabled: false                      # disabled until needed

  minimax:
    api_key: "${MINIMAX_API_KEY}"
    enabled: false

  kimi:
    api_key: "${KIMI_API_KEY}"
    enabled: false

  qwen:
    api_key: "${QWEN_API_KEY}"
    enabled: false

  # ── OpenAI-Compatible (custom endpoint) ──────────
  openai_compatible:
    api_key: "${CUSTOM_API_KEY}"
    endpoint: "https://my-company-proxy.example.com/v1"
    enabled: false
    default_model: "custom-model-name"

  # ── Local Provider ───────────────────────────────
  ollama:
    endpoint: "http://localhost:11434"
    enabled: true
    default_model: "llama3.1:8b"

  # ── Testing ──────────────────────────────────────
  mock:
    enabled: false                      # for testing only

# ── Routing Configuration ──────────────────────────
routing:
  strategy: "balanced"
  fallback_chain:
    - anthropic
    - openai
    - ollama

# ── Token Limits ───────────────────────────────────
tokens:
  max_input: 100000
  max_output: 8192
  warning_threshold: 0.8               # warn at 80% of limits
📝 Note

📝 Note: The "${ENV_VAR}" syntax lets you reference environment variables directly in the YAML file. This keeps secrets out of the config file while still being explicit about which variables are expected.

12.8 Compound AI — Multiple Model Roles (v0.3.5.1) #

v0.3.5.1 introduces a compound AI model system where different models serve different roles within a single session. Instead of one model doing everything, NeamCode assigns the right model to the right task:

Role Purpose Default Model
Normal Standard reasoning and coding Your primary provider
Thinking Extended thinking with deep reasoning Same as Normal (with thinking budget)
Critique Quality evaluation and plan review A fast, capable model
Fast Quick decisions (continue/complete/escalate) Smallest capable model
VLM Vision and multimodal tasks Multimodal-capable model

How It Works in Practice #

You don't need to configure this manually — NeamCode handles role assignment automatically:

text
You: Refactor the payment module to use the Strategy pattern

NeamCode internally:
  [Fast model]    → Classify task complexity: "multi-step refactor"
  [Thinking model] → Deep reasoning about Strategy pattern design
  [Critique model] → Evaluate the proposed approach
  [Normal model]   → Generate the refactored code
  [Fast model]    → Decide: continue to tests or complete?

Fallback Chains per Role #

Each role has its own fallback chain. If your Thinking model is unavailable, it falls back independently without affecting other roles:

yaml
# In neamcode.config.yml (optional — defaults are smart)
routing:
  strategy: "balanced"
  fallback_chain:
    - openai
    - anthropic
    - ollama
💡 Tip

💡 Tip: The compound AI system is why v0.3.5.1 achieves better results on complex tasks — it uses the right model for each sub-task rather than forcing one model to do everything. You don't need to configure it; just set up multiple providers and NeamCode optimizes automatically.

12.9 Pro Tips #

💡 Tip

💡 Tip: Start with two providers. Configure Anthropic (or OpenAI) for cloud quality and Ollama for free local work. This covers 90% of use cases and gives you a fallback chain.

💡 Tip

💡 Tip: Use /models --tier fast for quick tasks. Don't burn premium tokens on "what's the syntax for a for loop?" questions. Let the routing engine handle this by using the balanced or cost strategy.

💡 Tip

💡 Tip: The OpenAI-compatible provider is your escape hatch. If your company runs a proxy, or you want to use a provider not on the built-in list, the openai_compatible provider accepts any OpenAI-compatible API endpoint.

💡 Tip

💡 Tip: Test your fallback chain. Temporarily disable your primary provider in the config and confirm that NeamCode gracefully falls back. Better to discover issues during testing than during a real outage.

💡 Tip

💡 Tip: Gemini for giant contexts. If you need to analyze an entire repository or a massive log file, route to Gemini explicitly — its 1M+ token context window is unmatched.

⚠️ Warning

⚠️ Warning: Each provider has different rate limits. If you're hitting rate limits frequently, add a second provider to your fallback chain rather than waiting for cooldowns.
