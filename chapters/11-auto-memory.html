"The true art of memory is the art of attention."
— Samuel Johnson

11.1 What Gets Remembered #

NeamCode's Auto Memory (introduced in v0.3.5) observes your conversations and automatically stores important facts about your project. You don't need to configure anything — it learns as you work.

Here's what Auto Memory captures:

text
┌──────────────────────────────────────────────────────┐
│          WHAT AUTO MEMORY RECORDS                     │
│                                                      │
│  ✓ Technology choices                                │
│    "We use PostgreSQL with pgvector"                 │
│    "Testing framework is Vitest"                     │
│                                                      │
│  ✓ Architecture decisions                            │
│    "Auth is handled by Clerk"                        │
│    "API follows REST, not GraphQL"                   │
│                                                      │
│  ✓ Team conventions                                  │
│    "We prefer explicit error handling"               │
│    "All new endpoints need rate limiting"            │
│                                                      │
│  ✓ Deployment details                                │
│    "Staging deploys to AWS ECS Fargate"              │
│    "Production uses blue-green deployment"           │
│                                                      │
│  ✓ Bug patterns and fixes                            │
│    "WebSocket pool has connection leak on errors"    │
│    "Use idempotency keys for Stripe webhooks"        │
│                                                      │
│  ✓ Explicit instructions from you                    │
│    "Remember that we use 2-space indentation"        │
│    "Remember that the admin panel is at /internal"   │
│                                                      │
└──────────────────────────────────────────────────────┘

Auto Memory works in the background. When NeamCode notices a fact that seems important and project-specific, it adds it to your MEMORY.md file. When you start a new session in the same project, those facts are loaded automatically.

The result: NeamCode gets smarter about your project the more you use it.

11.2 MEMORY.md Explained #

MEMORY.md is a plain-text file where Auto Memory stores what it learns. It lives in your NeamCode configuration directory, isolated per project.

Location #

text
~/.neamcode/memory/<project-hash>/MEMORY.md

The <project-hash> is a SHA-256 hash of your project's root path, ensuring complete isolation between projects (more on this in Section 11.5).

Format #

MEMORY.md is simple and human-readable:

markdown
# Project Memory

- Project uses PostgreSQL 15 with pgvector extension
- Auth is handled by Clerk (clerk.com)
- API follows REST conventions, not GraphQL
- Deploy target is AWS ECS Fargate
- Testing framework is Vitest, not Jest
- Team prefers explicit error handling over try/catch
- All API endpoints must include rate limiting
- Frontend uses React 19 with Next.js 15
- CSS framework is Tailwind v4
- The admin panel is accessible at /internal/admin
- Database migrations use Drizzle ORM
- CI/CD runs on GitHub Actions
- Staging environment: staging.myapp.com
- Production environment: app.myapp.com

The 200-Line Cap #

MEMORY.md has a hard limit of 200 lines. This is intentional — it forces memory to stay focused and prevents it from consuming too much context.

When MEMORY.md approaches the cap, NeamCode:

  1. Groups related entries into topic overflow files
  2. Keeps only a summary line in MEMORY.md
  3. Loads topic files on demand when relevant
text
MEMORY.md (main file, < 200 lines)
├── General project facts
├── Key conventions
└── Summary pointers to topic files

Topic overflow files:
├── deployment.md      (detailed deployment knowledge)
├── database.md        (database schema and patterns)
└── api-conventions.md (API design rules)

Viewing Memory #

text
neamcode> /memory auto show

  MEMORY.md Contents
  ═══════════════════
  - Project uses PostgreSQL 15 with pgvector extension
  - Auth is handled by Clerk (clerk.com)
  - API follows REST conventions, not GraphQL
  - Deploy target is AWS ECS Fargate
  ...
  (14 entries)
text
neamcode> /memory auto topics

  Topic Overflow Files
  ════════════════════
  1. deployment.md      (42 lines)
  2. database.md        (28 lines)
  3. api-conventions.md (35 lines)

11.3 Telling NeamCode to Remember #

While Auto Memory captures many things on its own, you can explicitly tell NeamCode to remember something.

The Natural Way #

Just say it in plain language:

text
neamcode> Remember that we use PostgreSQL 15 with the pgvector extension

  ✓ Saved to MEMORY.md:
  "Project uses PostgreSQL 15 with pgvector extension"
text
neamcode> Remember that all new API endpoints need rate limiting
  middleware applied

  ✓ Saved to MEMORY.md:
  "All new API endpoints need rate limiting middleware applied"
text
neamcode> Remember: staging deploys go through the deploy-staging
  GitHub Action workflow, not manual kubectl

  ✓ Saved to MEMORY.md:
  "Staging deploys use deploy-staging GitHub Action workflow, not manual kubectl"

What to Tell It to Remember #

Focus on things that are:

  • Not obvious from the code itself
  • Important conventions your team follows
  • Decisions that took time to make (so you don't re-debate them)
  • Environment-specific details that aren't in config files
text
Good things to remember:
  "We use 2-space indentation in all TypeScript files"
  "The legacy API is at /api/v1 — don't modify it"
  "Authentication tokens expire after 24 hours"
  "The CEO's demo account is user ID 42 — never delete it"

Less useful things to remember:
  "The project is written in TypeScript"  (obvious from files)
  "npm install installs dependencies"     (universal knowledge)
💡 Tip

💡 Tip: When you finish a debugging session and find a non-obvious root cause, tell NeamCode to remember it. Next time someone hits the same issue, NeamCode will already know the answer.

11.4 Telling NeamCode to Forget #

Memory can be wrong or outdated. You can remove entries just as easily as you add them.

Forget the Last Thing #

text
neamcode> Remember that we use MySQL for the database

  ✓ Saved to MEMORY.md:
  "Project uses MySQL for the database"

neamcode> Wait, that's wrong. Forget that.

  ✓ Removed from MEMORY.md:
  "Project uses MySQL for the database"

Forget Something Specific #

text
neamcode> Forget that we use Tailwind v3

  ✓ Removed from MEMORY.md:
  "CSS framework is Tailwind v3"

Clear All Memory #

If you want to start fresh:

text
neamcode> /memory auto clear

  ⚠ This will delete all entries in MEMORY.md and topic files.
  Continue? (y/n) y

  ✓ Cleared MEMORY.md (14 entries removed)
  ✓ Cleared 3 topic overflow files
⚠️ Warning

⚠️ Warning: /memory auto clear is irreversible. The entries are deleted, not archived. If you're unsure, use /memory auto show first to review what's there.

Check Memory Status #

text
neamcode> /memory auto

  Auto Memory: enabled
  Location: ~/.neamcode/memory/a3f8b2c1/MEMORY.md
  Entries: 14 (of 200 line cap)
  Topics: 3 overflow files
  Last updated: 2 hours ago

11.5 Project-Specific Memory #

One of Auto Memory's most important features is project isolation. NeamCode maintains completely separate memory for each project.

How Isolation Works #

text
┌─────────────────────────────────────────────────────────┐
│              PROJECT MEMORY ISOLATION                     │
│                                                         │
│  ~/projects/webapp/                                     │
│  └── SHA-256 hash: a3f8b2c1...                          │
│      └── MEMORY.md: "Uses PostgreSQL, React, Tailwind"  │
│                                                         │
│  ~/projects/mobile-app/                                 │
│  └── SHA-256 hash: 7e2d9f04...                          │
│      └── MEMORY.md: "Uses SQLite, React Native, Expo"   │
│                                                         │
│  ~/projects/data-pipeline/                              │
│  └── SHA-256 hash: b1c4a890...                          │
│      └── MEMORY.md: "Uses DuckDB, Python, Airflow"      │
│                                                         │
│  Each project gets its own MEMORY.md.                   │
│  No cross-contamination.                                │
│                                                         │
└─────────────────────────────────────────────────────────┘

The isolation key is a SHA-256 hash of the project's absolute root path. This means:

  • /Users/alice/projects/webapp and /Users/bob/projects/webapp get different memory
  • Moving a project to a new directory creates a fresh memory space
  • Renaming a project directory creates a fresh memory space

Why Isolation Matters #

Without isolation, NeamCode might tell you to "use PostgreSQL" when you're working on a project that uses MongoDB. Project isolation ensures that memories from one project never leak into another.

text
# In ~/projects/webapp
neamcode> What database do we use?

  From memory: This project uses PostgreSQL 15 with pgvector.

# In ~/projects/mobile-app
neamcode> What database do we use?

  From memory: This project uses SQLite with Expo's built-in driver.
📝 Note

📝 Note: If you move your project to a new directory, NeamCode will start with a fresh MEMORY.md. You can migrate the old memory using /memory auto migrate (covered in Section 11.10).

11.6 The NEAMCODE.md File #

While MEMORY.md stores facts NeamCode learns over time, NEAMCODE.md is where you write explicit, permanent instructions for how NeamCode should behave.

Think of the difference this way:

text
MEMORY.md    = "Things NeamCode learned" (automatic, editable)
NEAMCODE.md  = "Rules you set"          (manual, authoritative)

What Goes in NEAMCODE.md #

markdown
# NEAMCODE.md

## Code Style
- Use 2-space indentation
- Prefer `const` over `let`
- Always use TypeScript strict mode
- Write JSDoc comments for public functions

## Architecture
- Follow the repository pattern for data access
- All business logic goes in src/services/
- Controllers only handle HTTP request/response

## Testing
- Every new function needs at least one unit test
- Use Vitest with the testing-library for React components
- Test files live next to source files: `foo.ts` → `foo.test.ts`

## Deployment
- Never push directly to main
- All changes go through pull requests
- Staging deploys automatically on PR merge to develop

The 7-Tier Hierarchy #

NEAMCODE.md files can exist at multiple levels. NeamCode reads all of them and merges them together, with higher tiers taking priority.

text
TIER  LOCATION                              PRIORITY
════  ══════════════════════════════════     ════════
 7    CLI --config flag                      Highest
 6    ./.neamcode/NEAMCODE.md (local)           ↑
 5    ./subdir/NEAMCODE.md (subdirectory)       │
 4    ./NEAMCODE.md (project root)              │
 3    ../NEAMCODE.md (ancestor directories)     │
 2    ~/.neamcode/NEAMCODE.md (global)          │
 1    Managed/default NEAMCODE.md           Lowest

Let's walk through each tier:

Tier 1: Managed (Lowest Priority)

The built-in defaults that ship with NeamCode. You never edit these.

Tier 2: Global

Your personal preferences that apply to every project:

text
~/.neamcode/NEAMCODE.md
markdown
# Global NeamCode Config

- Always explain changes before making them
- Prefer functional programming patterns
- I use macOS with zsh
- My editor is VS Code

Tier 3: Ancestor Directories

Instructions in parent directories. Useful for monorepos:

text
~/projects/NEAMCODE.md           ← applies to all projects in ~/projects/
~/projects/webapp/NEAMCODE.md    ← applies specifically to webapp

Tier 4: Project Root

The most common location. This is where team-wide rules live:

text
./NEAMCODE.md     ← at the root of your git repository

This file is typically committed to git so the whole team shares the same instructions.

Tier 5: Subdirectory

Rules that only apply within a specific directory:

text
./src/api/NEAMCODE.md     ← only applies when working in src/api/
markdown
# API-Specific Rules
- All handlers must validate input with Zod schemas
- Return standardized error objects from src/errors/
- Include request ID in all log messages

Tier 6: Local (.neamcode directory)

Personal overrides that you don't commit to git:

text
./.neamcode/NEAMCODE.md

Add .neamcode/ to your .gitignore for personal customizations.

Tier 7: CLI Config (Highest Priority)

Passed via the --config flag for one-off overrides:

bash
neamcode --config /path/to/custom-config.md

How Merging Works #

Higher tiers override lower tiers. If your global config says "use 4-space indentation" but your project config says "use 2-space indentation," the project config wins.

text
Global (Tier 2):    "Use 4-space indentation"
Project (Tier 4):   "Use 2-space indentation"
                     ↓
Result:             "Use 2-space indentation" (Tier 4 wins)

Instructions that don't conflict are additive — NeamCode follows all of them.

💡 Tip

💡 Tip: Put team conventions in the project-root NEAMCODE.md (Tier 4) and commit it to git. Put personal preferences in ~/.neamcode/NEAMCODE.md (Tier 2). Put directory-specific rules in subdirectory files (Tier 5).

11.7 @path Imports #

NEAMCODE.md files support an @path import syntax for reusing shared configuration across projects.

Basic Import #

markdown
# NEAMCODE.md

@path ~/.neamcode/shared/typescript-rules.md
@path ~/.neamcode/shared/testing-rules.md

## Project-Specific Rules
- Use Drizzle ORM for database access
- All new tables need migration files

The imported files are inlined at their position, as if you had copy-pasted their contents.

What You Can Import #

text
@path ~/shared-configs/company-standards.md     ← absolute path
@path ./.neamcode/team-rules.md                 ← relative path
@path ../shared/monorepo-rules.md               ← parent directory

Shared Configuration Pattern #

For organizations with many projects that share conventions:

text
~/.neamcode/shared/
├── code-style.md          ← company-wide code style
├── security-rules.md      ← security requirements
├── testing-standards.md   ← testing conventions
└── deploy-checklist.md    ← deployment rules

Then in each project's NEAMCODE.md:

markdown
# NEAMCODE.md

@path ~/.neamcode/shared/code-style.md
@path ~/.neamcode/shared/security-rules.md
@path ~/.neamcode/shared/testing-standards.md

## This Project
- Uses PostgreSQL 15
- Frontend is Next.js 15

Import Resolution #

Imports are resolved at load time. If an imported file doesn't exist, NeamCode logs a warning and continues:

text
  ⚠ NEAMCODE.md: Could not resolve @path ~/.neamcode/shared/missing.md
  Continuing without this import.
📝 Note

📝 Note: Imports do not support circular references. If file A imports file B and file B imports file A, NeamCode will detect the cycle and stop.

11.8 The Rules Engine #

For more advanced control, NeamCode includes a rules engine that lets you define path-scoped rules with YAML frontmatter.

Rules Directory #

text
.neamcode/rules/
├── api-standards.md
├── frontend-patterns.md
├── database-rules.md
└── security-checks.md

Rule File Format #

Each rule file is a Markdown file with optional YAML frontmatter:

markdown
---
paths:
  - "src/api/**"
  - "src/routes/**"
---

# API Standards

- All endpoints must validate input using Zod schemas
- Return proper HTTP status codes (don't use 200 for errors)
- Include pagination for list endpoints
- Rate limit all public endpoints
- Log request duration for performance monitoring

How Path Scoping Works #

The paths: frontmatter restricts when the rule is active. The rule only applies when NeamCode is working with files matching those glob patterns.

markdown
---
paths:
  - "src/api/**"
---
# Only active when working in src/api/
markdown
---
paths:
  - "**/*.test.ts"
  - "**/*.spec.ts"
---
# Only active when working with test files
markdown
# No frontmatter — this rule applies everywhere

Rules in Action #

text
neamcode> Write a new endpoint for /api/products

  Applying rules: api-standards.md (matched src/api/**)

  Creating src/api/products.ts with:
  ✓ Zod input validation schema
  ✓ Proper HTTP status codes
  ✓ Pagination for list endpoint
  ✓ Rate limiting middleware
  ✓ Request duration logging

NeamCode automatically loads and applies matching rules based on the files involved in the current task. You don't need to reference rules manually.

Multiple Matching Rules #

When multiple rules match, they all apply. If rules conflict, the more specific path wins:

text
.neamcode/rules/general.md          ← paths: ["src/**"]
.neamcode/rules/api-specific.md     ← paths: ["src/api/**"]

When working in src/api/, both rules apply. If they conflict, api-specific.md takes priority because its path is more specific.

💡 Tip

💡 Tip: Start with one or two broad rules and add specific ones as you discover patterns. Don't try to write every rule up front — let them evolve with your project.

11.9 Compaction Resilience #

A key design goal of Auto Memory is surviving context compaction (covered in Chapter 10, Section 10.6).

The Problem #

When NeamCode compacts a long conversation, it summarizes old messages. Important project facts could be lost in that summary.

The Solution #

MEMORY.md entries and NEAMCODE.md rules are never part of the conversation that gets compacted. They exist outside the conversation context:

text
┌──────────────────────────────────────────────────────┐
│              WHAT GETS COMPACTED                      │
│                                                      │
│  ┌─────────────────────────────────────────┐         │
│  │   Conversation messages  ← COMPACTED    │         │
│  └─────────────────────────────────────────┘         │
│                                                      │
│  ┌─────────────────────────────────────────┐         │
│  │   MEMORY.md              ← NEVER        │         │
│  └─────────────────────────────────────────┘         │
│                                                      │
│  ┌─────────────────────────────────────────┐         │
│  │   NEAMCODE.md            ← NEVER        │         │
│  └─────────────────────────────────────────┘         │
│                                                      │
│  ┌─────────────────────────────────────────┐         │
│  │   Rules (.neamcode/rules) ← NEVER       │         │
│  └─────────────────────────────────────────┘         │
│                                                      │
│  Memory and rules are reloaded fresh every time,     │
│  independent of conversation history.                │
│                                                      │
└──────────────────────────────────────────────────────┘

This means:

  • After compaction, NeamCode still knows your PostgreSQL version, your team conventions, and your deployment target
  • Even if you /clear the conversation entirely, memory persists
  • Starting a brand new session in the same project loads all existing memory

The Memory + Remember Pattern #

If you discover something important mid-conversation and want to ensure it survives any future compaction:

text
neamcode> (after a long investigation)
  So the root cause of the memory leak is that WebSocket
  connections in src/ws/pool.ts aren't released on error
  because the error handler on line 142 returns early
  before calling pool.release().

neamcode> Remember that the WebSocket memory leak is caused by
  the error handler in src/ws/pool.ts line 142 returning early
  before calling pool.release()

  ✓ Saved to MEMORY.md

Now even if this session is compacted or cleared, the critical finding is preserved.

11.10 Migrating from Older Versions #

If you've been using NeamCode before v0.3.5, you may have memory or configuration from older formats.

The /memory auto migrate Command #

text
neamcode> /memory auto migrate

  Migration Check
  ═══════════════

  Found legacy memory files:
  • ~/.neamcode/memory.json (v0.2.x format)
  • ./CLAUDE.md (v0.1.x-style instructions)

  Migration plan:
  1. Convert memory.json entries → MEMORY.md format
  2. Import relevant CLAUDE.md rules → NEAMCODE.md

  Proceed? (y/n) y

  ✓ Migrated 8 entries from memory.json → MEMORY.md
  ✓ Imported 5 rules from CLAUDE.md → NEAMCODE.md
  ✓ Legacy files preserved (not deleted)

What Gets Migrated #

Old Format New Format Notes
memory.json MEMORY.md Entries converted to bullet points
CLAUDE.md instructions NEAMCODE.md Rules imported, reviewed for relevance
.claude/ directory .neamcode/ directory Structure preserved
Old session format New session format Automatic on /resume

Manual Migration #

If you prefer to migrate manually:

  1. Create NEAMCODE.md at your project root
  2. Copy relevant rules from any existing CLAUDE.md
  3. Review ~/.neamcode/memory/ for any entries to bring forward
  4. Use /memory auto show to verify the new state
📝 Note

📝 Note: Migration is non-destructive. Your old files are preserved. The new system reads from the new locations, so old files simply become inactive backups.

11.11 Pro Tips #

Memory Audit #

Periodically review what NeamCode remembers. Outdated memories lead to outdated suggestions:

text
neamcode> /memory auto show

  (review each entry)

neamcode> Forget that we use Tailwind v3
  ✓ Removed

neamcode> Remember that we upgraded to Tailwind v4
  ✓ Saved

The NEAMCODE.md Starter Template #

Here's a solid starting point for a new project's NEAMCODE.md:

markdown
# NEAMCODE.md

## Project Overview
- Brief description of what this project does
- Key technologies: [list them]

## Code Conventions
- Indentation: [2 spaces / 4 spaces / tabs]
- Language: [TypeScript strict / Python 3.12 / etc.]
- Style guide: [link or name]

## Architecture
- Where business logic lives
- Where data access happens
- Key patterns used (MVC, repository, CQRS, etc.)

## Testing
- Framework: [Vitest / pytest / etc.]
- Test file location convention
- Minimum coverage expectations

## Things to Avoid
- [List common mistakes or forbidden patterns]

Commit your NEAMCODE.md to git. When a new team member clones the repo and starts NeamCode, they get all the team's conventions instantly:

bash

$ neamcode

  1. Create the route in src/api/
  2. Add Zod validation schema
  3. Use the repository pattern for data access
  4. Add rate limiting middleware
  5. Write tests in a co-located .test.ts file
  ...

Here's how all the pieces fit together:

text

               ┌───────────────┼───────────────┐

Debugging Memory Issues #

If NeamCode seems to "forget" things or behaves unexpectedly:

text
neamcode> /memory auto            # Check if memory is enabled
neamcode> /memory auto show       # See what's stored
neamcode> /memory auto topics     # Check topic overflow files
neamcode> /doctor                 # Verify NEAMCODE.md is found
neamcode> /status                 # Check overall state

Common issues:

  • Wrong directory: NeamCode uses your cwd to determine the project. If you're in a subdirectory, check that the project root is detected correctly.
  • Stale memory: Use "forget that" or /memory auto clear to remove outdated entries.
  • Conflicting rules: Check your 7-tier hierarchy. Higher tiers override lower ones.
