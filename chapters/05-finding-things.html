Chapter 5: Finding Things in Code #

"A codebase is a city. You need to know how to navigate it before you can build in it."
— The NeamCode Handbook

📖 Estimated read time: 18 minutes

What You'll Learn #

  • How to find files by name using glob patterns
  • How to search inside files for specific text or patterns with grep
  • How to visualize project structure with dir_tree
  • How to analyze code architecture with code_structure
  • A complete walkthrough of exploring an unfamiliar codebase

5.1 Finding Files by Name (glob) #

The glob skill searches for files by their filename pattern. Think of it as a smarter version of find that understands wildcards.

Basic Glob Searches #

text
You: Find all TypeScript files in the project

NeamCode:
  [glob] **/*.ts

  Found 47 files:
  src/index.ts
  src/app.ts
  src/config/database.ts
  src/config/auth.ts
  src/controllers/userController.ts
  src/controllers/postController.ts
  src/models/User.ts
  src/models/Post.ts
  src/routes/users.ts
  src/routes/posts.ts
  src/middleware/auth.ts
  src/middleware/validation.ts
  ... (35 more)

Common Glob Patterns #

Glob Pattern Quick Reference
PatternMatches
*.tsTypeScript files (current dir)
**/*.tsTypeScript files (all dirs)
src/**/*.test.tsTest files under src/
**/config.*Any config file, any extension
**/*.{ts,tsx}Both .ts and .tsx files
**/README*All README files
src/models/*.tsModels only (no subdirectories)

Practical Examples #

Finding config files:

text
You: Where are the config files in this project?

NeamCode:

  Found 6 files:
  .eslintrc.config.js
  jest.config.ts

Finding test files:

text
You: Show me all test files

NeamCode:
  [glob] **/*.test.{ts,tsx,js,jsx}
  [glob] **/*.spec.{ts,tsx,js,jsx}
  [glob] **/__tests__/**

  Found 23 test files across the project:
  tests/unit/user.test.ts
  tests/unit/post.test.ts
  tests/integration/api.test.ts
  src/components/__tests__/Header.test.tsx
  ...

Finding a specific file when you know part of the name:

text
You: There's a file with "migration" in the name somewhere

NeamCode:
  [glob] **/*migration*

  Found 3 files:
  db/migrations/001_create_users.sql
  db/migrations/002_create_posts.sql
  scripts/run-migrations.sh
💡 Tip

💡 Tip: glob searches by filename and path. If you need to search for text inside files, use grep (next section). The two skills complement each other perfectly.

5.2 Searching Inside Files (grep) #

The grep skill searches for text content within files. It's your go-to when you know what code says but not where it lives.

Basic Text Search #

text
You: Where is the database connection string defined?

NeamCode:
  [grep] "DATABASE_URL" across all files

  Found in 3 files:

  .env (line 4):
    DATABASE_URL=postgres://localhost:5432/myapp

  src/config/database.ts (line 7):
    const url = process.env.DATABASE_URL;

  docker-compose.yml (line 12):
    DATABASE_URL: postgres://db:5432/myapp

Searching for Function Definitions #

text
You: Where is the validateEmail function defined?

NeamCode:
  [grep] "function validateEmail" across all files
  [grep] "validateEmail =" across all files
  [grep] "const validateEmail" across all files

  Found definition in src/utils/validation.ts (line 23):
    export const validateEmail = (email: string): boolean => {

  Used in 4 other files:
  src/controllers/userController.ts (line 15): import
  src/controllers/authController.ts (line 8): import
  src/routes/users.ts (line 31): called
  tests/unit/validation.test.ts (line 12): tested

Searching Specific File Types #

text

📝 Note

📝 Note: NeamCode often combines grep with file_read automatically. It will search to find the right file, then read it to understand the full context before answering your question.

Understanding Architecture from Structure #

text
You: Based on the project structure, what architecture
     pattern is this project using?

NeamCode:
  [dir_tree] src/

  This project follows the **MVC (Model-View-Controller)** pattern:

  - models/     → Data layer (Mongoose/Sequelize models)
  - controllers/ → Business logic
  - routes/     → HTTP routing (acts as the "View" for an API)
  - middleware/  → Cross-cutting concerns
  - services/   → External integrations
  - utils/      → Shared helpers

  It's a clean separation. The only thing I'd suggest is adding
  a `types/` directory for shared TypeScript interfaces.
💡 Tip

💡 Tip: Running dir_tree is usually the first thing experienced NeamCode users do when opening a new project. It gives you a mental map before diving into any code.

5.4 Analyzing Code Structure (code_structure) #

The code_structure skill goes deeper than dir_tree. It analyzes the actual code to show you classes, functions, exports, and relationships.

Analyzing a File's Structure #

text
You: What's the structure of src/services/api.ts?

NeamCode:
  [code_structure] src/services/api.ts

  src/services/api.ts
  ├── Imports
  │   ├── axios (external)
  │   └── ApiConfig from ../config
  ├── Types
  │   ├── interface ApiResponse<T>
  │   └── type HttpMethod = 'GET' | 'POST' | 'PUT' | 'DELETE'
  ├── Class: ApiClient
  │   ├── private baseUrl: string
  │   ├── private headers: Record<string, string>
  │   ├── constructor(config: ApiConfig)
  │   ├── async get<T>(path: string): Promise<ApiResponse<T>>
  │   ├── async post<T>(path: string, data: unknown): Promise<ApiResponse<T>>
  │   ├── async put<T>(path: string, data: unknown): Promise<ApiResponse<T>>
  │   ├── async delete(path: string): Promise<void>
  │   └── private handleError(error: AxiosError): never
  └── Exports
      └── default ApiClient

Analyzing a Module or Directory #

text
You: Show me the code structure of the entire models directory

NeamCode:
  [code_structure] src/models/

  src/models/
  ├── User.ts
  │   ├── interface IUser
  │   ├── class User extends Model
  │   │   ├── static findByEmail(email: string)
  │   │   ├── async hashPassword()
  │   │   └── async comparePassword(candidate: string)
  │   └── exports: User, IUser
  ├── Post.ts
  │   ├── interface IPost
  │   ├── class Post extends Model
  │   │   ├── static findByAuthor(authorId: string)
  │   │   ├── static findPublished()
  │   │   └── async publish()
  │   └── exports: Post, IPost
  └── index.ts
      └── re-exports: User, Post, IUser, IPost

Finding Dependencies Between Modules #

text
You: What does the auth middleware depend on?

NeamCode:
  [code_structure] src/middleware/auth.ts
  [grep] "import" in src/middleware/auth.ts

  The auth middleware depends on:

  1. jsonwebtoken (external) — JWT verification
  2. ../models/User — User model for lookup
  3. ../config/auth — Secret keys and token config
  4. ../utils/errors — Custom error classes

  Dependency chain:
  auth.ts → User model → database config → .env
            → auth config → .env
            → jsonwebtoken (npm)

5.5 Exploring an Unfamiliar Codebase #

Let's walk through a realistic scenario: you've just cloned a repository and need to understand it quickly.

Step 1: Get the Lay of the Land #

text
You: I just cloned this repo. Help me understand it.

NeamCode:
  [dir_tree] .
  [file_read] package.json
  [file_read] README.md

  This is a **Next.js 14** application — an e-commerce platform.

  Key facts:
  - Framework: Next.js 14 with App Router
  - Language: TypeScript
  - Database: PostgreSQL via Prisma ORM
  - Auth: NextAuth.js
  - Styling: Tailwind CSS
  - Testing: Jest + React Testing Library
  - 47 source files, ~8,200 lines of code

Step 2: Understand the Architecture #

text
You: Show me how the code is organized

NeamCode:
  [dir_tree] src/
  [code_structure] src/app/

  The project uses Next.js App Router structure:

  src/
  ├── app/                    # Routes & pages
  │   ├── (auth)/             # Auth-grouped routes
  │   │   ├── login/
  │   │   └── register/
  │   ├── (shop)/             # Shop-grouped routes
  │   │   ├── products/
  │   │   ├── cart/
  │   │   └── checkout/
  │   ├── api/                # API routes
  │   │   ├── products/
  │   │   ├── orders/
  │   │   └── auth/
  │   ├── layout.tsx
  │   └── page.tsx
  ├── components/             # Reusable UI
  ├── lib/                    # Utilities & config
  ├── prisma/                 # Database schema
  └── types/                  # TypeScript types

  It follows the standard Next.js App Router conventions
  with route groups for auth and shop sections.

Step 3: Find the Key Files #

text
You: Where's the database schema?

NeamCode:
  [glob] **/*schema*
  [file_read] prisma/schema.prisma

  The Prisma schema defines 5 models:
  - User (id, email, name, password, role, orders)
  - Product (id, name, description, price, stock, category)
  - Category (id, name, products)
  - Order (id, userId, items, status, total, createdAt)
  - OrderItem (id, orderId, productId, quantity, price)

Step 4: Trace a Feature #

text
You: How does the checkout flow work? Walk me through the code.

NeamCode:
  [file_read] src/app/(shop)/checkout/page.tsx
  [file_read] src/components/CheckoutForm.tsx
  [file_read] src/app/api/orders/route.ts
  [file_read] src/lib/stripe.ts

  Here's the checkout flow:

  1. User visits /checkout → loads CheckoutForm component
  2. CheckoutForm collects shipping info and renders Stripe Elements
  3. On submit, it calls POST /api/orders with cart data
  4. API route:
     a. Validates cart items against database stock
     b. Creates a Stripe PaymentIntent
     c. Creates an Order record (status: "pending")
     d. Returns the client secret for Stripe confirmation
  5. Frontend confirms payment with Stripe
  6. Stripe webhook updates order status to "paid"
Checkout Page → API Route
/api/order → Stripe Payment → Webhook Update
Prisma Create Order

Created during API route processing

Order Status → "paid"

Updated by Stripe webhook callback

Step 5: Find Potential Issues #

text
You: Are there any obvious issues or code smells?

NeamCode:
  [grep] "TODO" across all files
  [grep] "FIXME" across all files
  [grep] "console.log" in src/**/*.ts
  [grep] "any" in src/**/*.ts

  I found several concerns:

  1. **12 TODO/FIXME comments** — most are in the checkout flow
  2. **7 console.log statements** in production code
  3. **15 uses of `any` type** — reduces TypeScript's safety
  4. **No error boundary** in the app layout
  5. **Stock check is not atomic** — race condition possible
     during concurrent checkouts (src/app/api/orders/route.ts:34)

5.6 Pro Tips #

1. Combine Search Skills #

The most powerful searches combine multiple skills:

text
You: Find all React components that use the useAuth hook

# NeamCode will:
# 1. grep for "useAuth" to find usage
# 2. glob for *.tsx to narrow to components
# 3. code_structure on matches to understand context

2. Use dir_tree First, Then Drill Down #

text
# Start broad:
You: Show me the project structure

# Then narrow:
You: Show me what's in the services directory

# Then go deep:
You: Analyze the code structure of the PaymentService

3. Search for Patterns, Not Just Text #

text
You: Find all API endpoints in this Express app

# NeamCode will grep for router.get, router.post, app.get, etc.
# This is smarter than searching for a single string.

4. Let NeamCode Choose the Right Tool #

You don't need to specify which skill to use. Just describe what you're looking for:

text
# These all work — NeamCode picks the right skill:
You: Where is the login page?           → glob + file_read
You: Who calls the sendEmail function?  → grep
You: What's the project layout?         → dir_tree
You: How is the User class structured?  → code_structure

5. Bookmark Your Findings Mentally #

When exploring a codebase, build a mental map:

text
You: I need to understand the auth system. Start by finding
     all auth-related files, then explain how they connect.

# NeamCode will give you the full picture in one response.
# This is faster than exploring file by file.

Exercises #

🎯 Exercise 5.1: Navigate to any project on your machine. Ask NeamCode to show its directory structure, then find all configuration files (any file with "config" in the name or common config extensions like .yml, .json, .toml).

🎯 Exercise 5.2: In the same project, use grep through NeamCode to find all import statements that reference a specific module. Then ask how many files depend on that module.

🎯 Exercise 5.3: Clone a popular open-source repository you've never worked with (e.g., a small Express.js project from GitHub). Ask NeamCode to explore it and explain the architecture in under 2 minutes.

🎯 Exercise 5.4: Ask NeamCode to find all functions that are defined but never called in a project. This requires combining code_structure with grep — see if NeamCode can figure it out.

🎯 Exercise 5.5: Pick a feature in a project (e.g., "user registration") and ask NeamCode to trace it from the frontend to the database, identifying every file involved.
