Chapter 13: Offline Mode — AI Without Internet #

"The network is unreliable. Your coding assistant shouldn't be."
— NeamCode Design Principle

📖 Estimated read time: 18 minutes

What You'll Learn #

  • How NeamCode's zero-config offline auto-recovery works under the hood
  • How to install and configure Ollama for local AI
  • What changes (and what stays the same) when you go offline
  • How to read the status bar indicators
  • Manual control with /offline, /online, and /status
  • A full walkthrough of a session that transitions between online and offline seamlessly

13.1 How Offline Recovery Works #

Starting with v0.3.4, NeamCode includes zero-config offline auto-recovery. This means you don't have to do anything — if your internet drops, NeamCode detects it and switches to local models automatically.

Here's what happens behind the scenes:

text
┌─────────────────────────────────────────────────────────────┐
│                  Offline Recovery Pipeline                    │
│                                                             │
│  Normal operation (online)                                  │
│       │                                                     │
│       ▼                                                     │
│  TCP health probe ────── every 30 seconds ──────┐           │
│       │                                          │           │
│       ├── Success? → stay online                 │           │
│       │                                          │           │
│       └── Failure? → increment fail count        │           │
│              │                                   │           │
│              ▼                                   │           │
│     2 consecutive failures                       │           │
│              │                                   │           │
│              ▼                                   │           │
│     ┌─────────────────┐                          │           │
│     │  SWITCH OFFLINE  │ → Route to Ollama       │           │
│     └─────────────────┘                          │           │
│              │                                   │           │
│              ▼                                   │           │
│     Continue probing...                          │           │
│              │                                   │           │
│     3 consecutive successes + 60s dwell time     │           │
│              │                                   │           │
│              ▼                                   │           │
│     ┌─────────────────┐                          │           │
│     │  SWITCH ONLINE   │ → Restore cloud provider│           │
│     └─────────────────┘                          │           │
│                                                             │
└─────────────────────────────────────────────────────────────┘

Key Concepts #

TCP Health Probing — Every 30 seconds, NeamCode sends a lightweight TCP probe to your configured cloud provider. This is not a full API call — it costs nothing and adds no latency to your work.

Flap Protection — Networks can be unstable. A single failed probe doesn't trigger offline mode. NeamCode requires 2 consecutive failures before switching offline. Similarly, it requires 3 consecutive successes plus 60 seconds of stable connectivity before switching back online. This prevents rapid back-and-forth switching (flapping) on unstable connections.

Zero-Config — You don't need to enable this, configure thresholds, or set up any triggers. If Ollama is installed and running, offline recovery works automatically.

💡 Tip

💡 Tip: The only prerequisite is having Ollama installed with at least one model pulled. See section 13.2 for setup.

13.2 Setting Up Ollama #

Ollama is the local AI runtime that powers NeamCode's offline mode. You need it installed with at least one model pulled for offline recovery to work.

Quick Install and Model Pull #

bash
# macOS
brew install ollama

# Linux
curl -fsSL https://ollama.com/install.sh | sh

# Start the service and pull a model
ollama serve &
ollama pull qwen2.5-coder:7b

See Chapter 14 for detailed guidance on model selection by hardware tier (8GB / 16GB / 32GB), the hybrid 80/15/5 cost strategy, full model tier rankings, and cost comparisons between local and cloud providers.

Verify the Setup #

Once Ollama is running with a model pulled, confirm NeamCode detects it:

text
neamcode $ /status

Provider Status:
  ● Anthropic    — connected (claude-sonnet)
  ● Ollama       — connected (qwen2.5-coder:7b, local)

Offline Recovery: enabled
  Local models: qwen2.5-coder:7b
  Health probe: every 30s

13.3 What Changes When You're Offline #

When NeamCode switches to offline mode, most things work the same. Here's what changes and what doesn't:

text
┌──────────────────────────────────────────────────────────┐
│                 Online vs. Offline                        │
├──────────────────────┬──────────────┬────────────────────┤
│ Feature              │ Online       │ Offline            │
├──────────────────────┼──────────────┼────────────────────┤
│ Code generation      │ ✓            │ ✓ (local model)    │
│ Code explanation     │ ✓            │ ✓                  │
│ File editing         │ ✓            │ ✓                  │
│ Terminal commands    │ ✓            │ ✓                  │
│ Git operations       │ ✓            │ ✓                  │
│ Context window       │ 128K–1M+     │ 4K–32K (varies)   │
│ Response quality     │ Premium      │ Good               │
│ Response speed       │ Depends      │ Depends on HW      │
│ Cost                 │ Per-token    │ Free               │
│ Privacy              │ Data sent    │ Fully local        │
│ Web search           │ ✓            │ ✗                  │
│ External APIs        │ ✓            │ ✗                  │
└──────────────────────┴──────────────┴────────────────────┘

What Gets Better Offline #

  • Cost: Zero. Every token is free.
  • Privacy: Nothing leaves your machine. No data sent anywhere.
  • Latency: On good hardware, local models can be faster than cloud round-trips.

What Gets Worse Offline #

  • Context window: Local models typically support 4K-32K tokens vs. 128K-1M+ for cloud.
  • Complex reasoning: Premium cloud models (Claude Opus, GPT-4o) outperform local models on difficult tasks.
  • Web access: No web search or external API calls.
📝 Note

📝 Note: NeamCode does not notify you disruptively when switching modes. You'll see the status bar change color (see 13.4) and a brief inline note, but your workflow continues uninterrupted.

13.4 The Status Bar #

The status bar at the bottom of your NeamCode terminal shows your connectivity state at a glance:

text
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  (your working area)                                    │
│                                                         │
├─────────────────────────────────────────────────────────┤
│ ● claude-sonnet │ tokens: 1.2K/200K │ cost: $0.04 │ ↑ │
└─────────────────────────────────────────────────────────┘

Status Indicator Colors #

● Green (Online) — Connected to cloud provider. Everything working normally.

text
● claude-sonnet │ tokens: 1.2K/200K │ cost: $0.04

● Yellow (Degraded) — Connection is unstable. NeamCode detected a failure but hasn't switched offline yet (1 failure, waiting for the second to confirm).

text
● claude-sonnet │ tokens: 1.2K/200K │ cost: $0.04 │ ⚠ unstable

● Red (Offline) — No internet connection. Running on local Ollama models.

text
● qwen2.5-coder:7b (local) │ tokens: 856/8K │ cost: $0.00

The transition looks like this in practice:

text
Before:   ● claude-sonnet │ tokens: 1.2K/200K │ cost: $0.04
                ↓ (network drops)
During:   ● claude-sonnet │ ... │ ⚠ unstable
                ↓ (2nd failure confirmed)
After:    ● qwen2.5-coder:7b (local) │ tokens: 0/8K │ cost: $0.00

And when connectivity returns:

text
Before:   ● qwen2.5-coder:7b (local) │ tokens: 2.1K/8K │ cost: $0.00
                ↓ (3 successes + 60s dwell)
After:    ● claude-sonnet │ tokens: 1.2K/200K │ cost: $0.04
💡 Tip

💡 Tip: The status bar is always visible. A quick glance tells you whether you're burning cloud tokens or running for free locally.

13.5 Manual Control (/offline, /online, /status) #

While auto-recovery handles most situations, sometimes you want explicit control.

/offline — Force Offline Mode #

text
neamcode $ /offline

Switched to offline mode.
  Provider: Ollama (qwen2.5-coder:7b)
  Auto-recovery: paused (use /online to resume)

● qwen2.5-coder:7b (local) │ cost: $0.00

Use this when:

  • You're about to board a flight and want to ensure no cloud calls are attempted
  • You're working with sensitive code and want guaranteed local processing
  • You want to save money during a long session of simple tasks

/online — Force Online Mode #

text
neamcode $ /online

Switching to online mode...
  Checking Anthropic... connected ✓
  Provider: Anthropic (claude-sonnet)
  Auto-recovery: resumed

● claude-sonnet │ cost: $0.04

/status — Full Status Report #

text
neamcode $ /status

Connection Status: ONLINE
─────────────────────────────────────
Network:
  Cloud connectivity:  ● connected
  Last health probe:   2s ago (success)
  Probe interval:      30s
  Flap protection:     0 failures / 0 of 2 threshold

Active Provider:
  Provider:            Anthropic
  Model:               claude-sonnet
  Context window:      200K tokens

Local Fallback:
  Ollama:              ● running
  Available models:    qwen2.5-coder:7b, phi3
  Best available:      qwen2.5-coder:7b (tier 2)

Session:
  Mode:                auto (online)

📝 Note

📝 Note: When you use /offline manually, auto-recovery is paused. NeamCode won't switch back to cloud on its own — you must explicitly run /online to re-enable automatic switching.

13.6 Seamless Session Example #

Here's a realistic scenario showing how offline recovery works during a coding session. You're on a train, working on a Python project.

text
neamcode $ ● claude-sonnet │ online

You: Add error handling to the database connection in db.py

NeamCode: I'll add proper error handling with retry logic...
  [edits db.py with try/except, exponential backoff, logging]

You: Good. Now add unit tests for the error cases.

NeamCode: I'll create tests for each error scenario...
  [creates test_db.py with 6 test cases]

                    ─── Train enters a tunnel ───

  ● claude-sonnet │ ⚠ unstable

  ● qwen2.5-coder:7b (local) │ $0.00
  ℹ Switched to local model (network unavailable)

You: Run the tests and fix any failures.

NeamCode: Running the tests locally...
  $ python -m pytest test_db.py -v

  3 passed, 3 failed. Let me fix the failures...
  [edits test_db.py — fixes mock setup issues]

  $ python -m pytest test_db.py -v
  6 passed ✓

You: Add a connection pool with max 10 connections.

NeamCode: I'll add connection pooling to db.py...
  [edits db.py with connection pool implementation]

                    ─── Train exits the tunnel ───

  ● claude-sonnet │ online
  ℹ Cloud connectivity restored

You: Review the connection pool implementation for thread safety issues.

NeamCode: Looking at the implementation with fresh eyes...
  I see two potential thread safety issues:
  1. The pool counter isn't atomic...
  2. The connection dict needs a lock...
  [suggests fixes with threading.Lock]

Notice what happened:

  1. You never had to stop working
  2. The switch to local was automatic (after 2 failed probes)
  3. The switch back was automatic (after 3 successes + 60s dwell time)
  4. Your conversation context was preserved throughout
  5. The complex review task naturally ran on the cloud model (higher quality for analysis)
⚠️ Warning

⚠️ Warning: Conversation history is preserved when switching modes, but the local model has a smaller context window. If your conversation is very long, the local model may truncate earlier messages. Keep important context in recent messages.

13.7 Pro Tips #

💡 Tip

💡 Tip: Pre-pull models before traveling. If you know you'll be offline, pull multiple Ollama models in advance. NeamCode will automatically select the best available one.

bash
# Before your flight
ollama pull qwen2.5-coder:7b
ollama pull llama3.1:8b
ollama pull phi3
💡 Tip

💡 Tip: Use /offline proactively for privacy. Working on sensitive code? Force offline mode with /offline and no data leaves your machine — guaranteed.

💡 Tip

💡 Tip: Monitor with /status after reconnecting. After a period offline, run /status to confirm the cloud provider reconnected successfully and check if any failures occurred during the transition.

💡 Tip

💡 Tip: The 60-second dwell time is your friend. The dwell time prevents NeamCode from switching back to cloud on a brief WiFi blip. If you see the yellow indicator, don't worry — it means the system is working as designed, waiting to confirm stable connectivity.

💡 Tip

💡 Tip: Ollama startup time matters. If Ollama isn't already running when you go offline, there will be a delay while it loads the model into memory. Keep ollama serve running in the background for instant failover.

bash
# Run Ollama as a background service (macOS)
brew services start ollama

# Or on Linux with systemd
sudo systemctl enable ollama
sudo systemctl start ollama
💡 Tip

💡 Tip: Check model tier ranking. Not all local models are equal. Run /status to see which tier your best available model is. Tier 1-3 models handle most coding tasks well. Tier 6-8 models are better suited for simple questions and completions.

⚠️ Warning

⚠️ Warning: Offline mode only works for AI-powered features. If your task requires actual internet access (cloning a repo, installing packages, accessing a remote API), you still need connectivity for those operations.
